# AI 亮主 / 反主专项计划

这份文档专门记录“亮主 / 反主”阶段的 AI 分级实现方案，供后续开发时直接对照落地。

目标不是改规则，而是把当前“按档位比较大小”的实现，逐步升级成“按局势判断是否值得亮 / 反”的实现。

当前说明：

- 这份文档主要保留“亮主 / 反主专项策略”的分级目标与设计拆解，不重复维护中级搜索主线的里程碑状态。
- 搜索主线状态以 [ai-checklist.md](ai-checklist.md) 为准。
- 术语沿用 [ai-roadmap.md](ai-roadmap.md)。
- 当前代码状态可粗略视为：
  - `初级` 亮主 / 反主策略已稳定
  - `中级` 亮主 / 反主第一阶段已落地
  - `高级` 亮主 / 反主仍主要停留在设计目标阶段

## 1. 目标分层

### 初级

- 保持当前实现，不额外增加复杂判断。
- 核心定位：
- `规则合法`
- `优先亮当前能亮的最高档`
- `保留少量随机性`

### 中级

- 从“能不能亮”升级到“现在亮值不值”。
- 只使用：
- 自己手牌
- 当前亮主信息
- 已公开流程信息
- 不主动推断他人暗手。

### 高级

- 从“是否值得亮”升级到“亮 / 反本身也是信息战”。
- 在中级基础上，增加：
- 对他人亮主行为的推断
- 对后续反主链的预估
- 对整局计划的联动判断

## 2. 当前实现现状

当前代码的主流程主要在：

- [src/shared/game.js](../src/shared/game.js)

其中与亮主 / 反主最直接相关的入口为：

- `getDeclarationOptions`
- `getBestDeclarationForPlayer`
- `getDeclarationPriority`
- `canOverrideDeclaration`
- `maybeAutoDeclare`
- `getCounterDeclarationForPlayer`
- `startCounterTurn`

当前策略特点：

- 更偏“规则驱动”，而不是“局势驱动”。
- 会比较档位大小，但几乎不会比较同档位不同主种的战略价值。
- 会决定“能不能反”，但不会充分评估“反了之后是不是更赚”。
- 发牌中一旦能亮，通常会较早亮，缺少延迟亮主思路。

这套实现适合保留给 `初级`，但不足以支撑 `中级 / 高级`。

## 3. 初级策略

### 3.1 保留原则

- 初级继续使用当前方案。
- 不额外引入复杂评分器。
- 不做多步推演。
- 不做暗手推断。

### 3.2 行为定义

- 有合法亮主方案时，取当前最高档。
- 自动亮花色主前，再加一层轻量门槛：
- 该方案下的主牌总数必须至少达到 `10 张`
- 这里的主牌总数指 `常主 + 主花色牌`
- `亮无主` 时，至少需要 `5 张常主`。
- `3 张` 组合比 `2 张` 组合更积极亮。
- `反无主` 时，至少需要 `5 张常主`。
- `反无主` 不能只看常主总数；如果当前已亮花色主对自己明显更合适，初级 AI 会倾向不反。
- 这里的 `常主` 指级牌和大小王。
- `3 张亮主` 的一个直接好处，是档位更高，不容易被后续更低档方案反压，因此初级 AI 仍会更积极出手。
- 其他 `3 张反主` 仍可更积极。
- 其他情况允许保留当前概率式出手。

### 3.3 预期体感

- 像“会规则，但不会深算”的新手。
- 偶尔亮早、偶尔反早是可以接受的。

## 4. 中级策略

中级不需要猜别人具体有什么牌，但要开始判断“我是否适合现在做庄”。

### 4.1 核心目标

- 允许延迟亮主。
- 比较同档位主种优劣。
- 判断反主是否真的提高庄位价值。
- 为后续扣底、叫朋友 / 找朋友、保底预留结构。

### 4.2 建议新增能力

#### A. 延迟亮主

- 不再默认“刚满足条件就立刻亮”。
- 典型延迟场景：
- 已有 `2 张同花色级牌`，但仍可能摸成 `3 张`
- 已有 `2 张王`，但当前无主结构并不理想
- 当前只是勉强可亮，但手牌明显不适合做庄

建议规则：

- 若当前方案只是“最低可亮强度”，且预计后续可能升级为更高档，则允许延迟。
- 若当前方案已经非常强，例如 `3 张大王`，则不延迟。

#### B. 同档位主种比较

- 对同档位方案补“主种价值评分”。
- 例如同样都是 `2 张级牌` 时，不应只按数组顺序取第一个。

建议评分项：

- `trumpCount`
  该主种下自己的总主数
- `highTrumpCount`
  该主种下的高主控制力
- `shortSuitPotential`
  扣底后是否更容易形成短门将吃
- `sideControl`
  副牌 A/K 等控制牌保留情况
- `friendCallFit`
  是否更方便后续叫朋友、找朋友与回手

#### C. 反主收益判断

- 不是“能反就反”，而是判断：
- 反完后我是不是更像强庄
- 反完后底牌是否更好处理
- 反完后主控是否真的更强
- 反完后是否只是把庄抢来，但实际手牌更别扭

建议加入两个中级评分：

- `declareSeatValue`
  我做庄是否舒服
- `counterUpgradeValue`
  这次反主相比当前亮主是否明显提升了我方庄位质量

#### D. 无主适配判断

- 中级应该开始区分“这手更适合打无主还是有主”。

适合无主的常见特征：

- 王多
- 级牌控制强
- 各副花色有较平均的大牌控制
- 不依赖短门将吃

适合有主的常见特征：

- 某门主花色牌长
- 主里有明显控制力
- 副牌结构适合做短门

### 4.3 中级实现建议

建议新增一个轻量评分入口，例如：

```js
scoreDeclarationOptionForIntermediate(playerId, option)
scoreCounterOptionForIntermediate(playerId, option, currentDeclaration)
shouldDelayDeclarationForIntermediate(playerId, option)
```

建议流程：

1. 先枚举所有合法亮主 / 反主方案。
2. 对每个方案计算中级分。
3. 若最佳方案未达到阈值，则允许不亮 / 不反。
4. 若只是低价值两张方案，且存在明显升级空间，则允许延迟。

当前已落地的中级第一阶段：

- 自动亮主会在同档位方案里比较主牌数量、主控、副牌控制和短门潜力。
- 发牌中若只有低价值的两张方案，且后续还有明显摸牌空间，会先延迟亮主。
- 自动反主会比较“新方案相对当前亮主的适配提升”。
- 如果只是合法但提升太小的反主，中级会直接选择不反。

### 4.4 中级预期体感

- 不会一有牌就急着亮。
- 会开始选择“更好打”的主，而不是“恰好能亮”的主。
- 会出现“明明能反，但判断不值得所以不反”的行为。

## 5. 高级策略

高级阶段开始把亮主 / 反主视为信息博弈。

### 5.1 核心目标

- 预估后续反主链。
- 利用别人亮主 / 不亮主行为更新判断。
- 把亮主决策和整局计划联动。

### 5.2 建议新增能力

#### A. 反主链预估

- 高级不只判断“我能不能压当前亮主”。
- 还要判断：
- 我反完后，后位还有没有更大概率继续压我
- 我是不是在替别人抬庄
- 我是否应该保留更高档资源，等待更合适时机

建议新增评估：

- `counterStability`
  这次反主后，我最终坐稳庄位的概率倾向

#### B. 从行为更新信息

- 基于公开动作形成弱推断，而不是透视式判断。

例如：

- 某玩家到发牌后段仍未亮，说明部分档位大概率缺失
- 某玩家只亮两张不抢三张，说明可能保留更高档资源
- 某玩家敢反无主，说明其王张 / 级牌密度可能更高

建议把这些信息记录成轻量状态，例如：

```js
state.declarationRead = {
  2: { likelyNoPairLevel: false, likelyHasJokerPower: true },
  3: { likelySavingUpgrade: true },
}
```

这里不要求精确，只要求能影响高级评分。

#### C. 亮主与整局计划联动

- 高级不应把亮主视为独立动作，而应连到：
- 扣底质量
- 叫朋友 / 找朋友效率
- 末局保底能力

建议新增评估：

- `buryFit`
  该主种是否利于拿底后重构手牌
- `friendPlanFit`
  该主种是否方便后续叫朋友、找朋友并回手
- `endgameGuardFit`
  该主种是否更利于保底

#### D. 无主 / 有主的深层适配

- 高级应比中级更稳定地判断：
- “这手牌虽然能亮有主，但打无主更强”
- “这手牌虽能反无主，但实际有主更能发挥短门优势”

这部分不只是统计主数，而是综合：

- 王张质量
- 级牌控制
- 副牌控制
- 短门潜力
- 找朋友路线

### 5.3 高级实现建议

建议新增高级评分入口，例如：

```js
scoreDeclarationOptionForAdvanced(playerId, option)
scoreCounterOptionForAdvanced(playerId, option, currentDeclaration)
estimateCounterStability(playerId, option, currentDeclaration)
updateDeclarationReadFromAction(playerId, action)
```

高级流程建议：

1. 先沿用中级评分框架。
2. 额外叠加行为信息与反主链评估。
3. 允许出现“为防后手继续压制而放弃当前反主”的决策。

### 5.4 高级预期体感

- 会“藏”。
- 会“等”。
- 会判断“抢到庄”和“坐稳庄”不是一回事。
- 会让亮主 / 反主显得更像一轮信息战。

## 6. 推荐工程拆分

### 第一阶段

- 保持初级不动。
- 把当前逻辑明确收口成：
- `beginner` 使用旧逻辑
- `intermediate / advanced` 走新评分入口

### 第二阶段

- 新增中级亮主评分器。
- 新增中级反主评分器。
- 实现“低价值方案允许延迟亮主”。

### 第三阶段

- 为高级增加 declaration 行为读牌状态。
- 增加反主链稳定性评估。
- 增加亮主与扣底 / 叫朋友 / 找朋友联动评分。

## 7. 建议落点

优先建议修改这些函数附近的逻辑：

- [src/shared/game.js](../src/shared/game.js)

具体建议：

- 保留 `getDeclarationOptions`
- 保留 `getDeclarationPriority`
- 保留 `canOverrideDeclaration`
- 重写 `getBestDeclarationForPlayer`，让它按难度走不同评分
- 重写 `maybeAutoDeclare`，让中高级支持延迟亮主
- 重写 `getCounterDeclarationForPlayer`，让中高级不再只取“能压住的最大档”
- 调整 `startCounterTurn` 中 AI 是否反主的判定，让其改为基于评分阈值而不是简单随机

## 8. 建议测试补充

建议新增专门回归：

- 中级：已有 `2 张级牌`，但继续摸牌有明显升成 `3 张` 机会时，应允许延迟
- 中级：同档位两门可亮时，应优先选择主控更强的一门
- 中级：虽然能反，但反后结构更差时，应选择不反
- 高级：存在后位更大概率继续压庄时，应减少无意义反主
- 高级：对他人“亮两张 / 不继续抢亮”的行为，能够影响后续反主选择
- 高级：同一手牌在“有主 / 无主”之间能做出更稳定选择

## 9. 最终分层定义

- `初级`
  会按规则亮主和反主，允许简单随机
- `中级`
  会判断现在亮 / 反值不值，但不主动猜别人暗手
- `高级`
  会把亮主 / 反主当成信息战，并联动整局计划

这就是后续实现“亮主 / 反主 AI”时推荐遵循的分级基线。
