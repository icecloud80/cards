# 初级 AI 启发式规则记录

这份文档单独记录当前产品中 `初级 AI` 的启发式规则，目标是把“规则总表里的对外口径”和“代码里的实际取舍”整理成一份更适合持续维护的说明。

如果未来 `src/shared/game.js`、`src/shared/ai-beginner.js`、`src/shared/ai-shared.js` 里的 beginner 行为发生变化，应优先同步这份文档，再回写到面向玩家的规则总表。

## 1. 定位

- 核心定位：`规则合法 + 固定启发式 + 少量身份协同`
- 信息边界：只使用公开信息，不读取他人暗手，不做完整记牌
- 体感目标：像“懂规则、会顺手配牌、但不会深算后手”的新手
- 行为气质：重眼前顺手，不专门为后两轮做复杂布局

## 2. 总原则

- 所有动作先过规则合法性检查，再应用 heuristic。
- 初级不使用搜索，不做“如果我这手打了，后面两轮会怎样”的复杂前瞻。
- 初级只记已经公开暴露出来的断门信息，不记完整已出牌。
- 初级会额外记公开已结算的分数归属；阵营已明后，会判断“闲家后面剩余分是否还够到 120”。
- 身份协同只保留非常直白的套路，不引入中级那种局势推断。
- 牌面资源如何组织只影响视觉表现；无论当前使用经典逐张 SVG、`poker.png` 还是 `m_cards_sprite.svg`，初级 heuristic 都不能因此变化。
- `m_cards_sprite.svg` 后续即使继续微调 tile 对齐、缩放或留白口径，也只属于渲染资源修复，不能被初级 heuristic 当成任何牌力或局势信号。
- 即便 `m_cards_sprite.svg` 被重制成严格 `90x120`、无缝拼版的 sprite，初级 heuristic 也不能把这种资源尺寸口径变化当成任何额外信息。
- 即便 PC / mobile 的运行态整图牌面入口被统一收口到 `m_cards_sprite.svg`，初级 heuristic 也不能把这种配置整理当成任何额外信息。
- mobile 手牌区的花色标签是否显示张数，同样只属于 UI 密度调整；初级 heuristic 不能把“玩家界面上有没有计数”当成任何额外信息来源。
- mobile 顶部 `主 / 朋` 状态牌是用 sprite 还是单张 `img`，也只属于表现层实现细节；初级 heuristic 不能从这类渲染差异里获得任何额外信息。
- 手动托管的 `关闭 / 本局托管 / 跨局托管` 三态只影响“当前是否由 AI 代打”和“是否跨局保留代打”，不能改变初级 heuristic 本身。
- `重置本局` 图标只负责保留当前级别并把这局重新洗牌、重新发牌；它属于流程控制入口，不能改变初级 heuristic、难度定位或策略边界。
- mobile 设置菜单是否额外再放一个 `重置本局` 按钮，同样只属于入口去重，不是 beginner heuristic 的一部分。

## 3. 亮主 / 反主

### 3.1 自动亮主

- 花色主：只有当 `常主 + 主花色牌 >= 10 张` 时，初级才愿意自动亮该门主。
- 无主：只有当 `常主 >= 5 张` 时，初级才愿意自动亮无主。
- 三张亮主仍然更积极，因为它的档位更高，更不容易被更低档方案反压。

### 3.2 自动反主

- 初级不会自动反自己已经亮出的主。
- 反无主：只有当 `常主 >= 5 张` 时，初级才愿意自动反无主。
- 如果当前已经亮出的花色主对自己明显更合适，初级不会轻率把它反成无主。

## 4. 叫朋友

### 4.1 选牌目标

- 初级优先找 `副牌 A` 做朋友牌。
- 如果同时有多门可找的 `副牌 A`，优先选择自己当前 `最短的那一门`。
- 如果自己已经持有这门的第一张 `A`，会优先叫第二张；如果已经持有前两张 `A`，会优先叫第三张。
- 只有在没有合适 `副牌 A` 候选时，才回落到旧的宽松 heuristic。

### 4.2 设计意图

- 这不是为了做复杂推断，而是为了给未来的朋友一个明确而直白的信号：
- `这门会被我主动打短，可以拿来回手。`

## 5. 扣底

### 5.1 与叫朋友联动

- 初级扣底会围绕“最短副牌 A”这条线服务，而不是独立做另一套判断。
- 对选中的短门，初级会尽量保留：
- `A`
- `1 张同门单牌回手`

### 5.2 扣底倾向

- 为了把这门做成更明确的短门，初级会优先把其他杂牌扣到底里。
- 如果这门能被整理成“`A + 1 张单牌回手`”的结构，初级会优先保住这条回手线，而不是默认继续保留同门 `K`。
- 如果必须在“拆掉短门回手牌”和“埋低主”之间二选一，初级会优先保住这条短门线。
- 但埋底仍然必须遵守数量与总分上限规则。

## 6. 首发

### 6.1 通用顺序

- 首发优先级大体是：
- 强制站队
- 打家延迟型找朋友前置张
- 无主打家清大牌
- 防止庄家将吃的安全首发
- 闲家侧先兑现高张、再准备递牌
- 基于公开绝门信息的递牌
- 根据公开断门信息施压
- 顺手的结构牌
- 最低成本单张

### 6.2 找朋友相关首发

- 如果初级叫的是第二张或第三张 `A`，且这门已经被做成 `A + 1 张单牌回手`，默认先走这张单牌回手。
- 这样做的默认假设是：找朋友这门会尽快被走空，后续打家更容易靠毙牌重新拿回控制权。
- 如果这门没有形成明确的单张回手口，才回退到旧的“尽快先走自己手里的前置 `A`”。

### 6.3 递牌相关首发

- 如果闲家侧已经拿到首发、同伴已知、而这门在公开信息里还没有任何人绝门，初级会先把自己这门的 `A` 或以 `A` 为顶张的大结构牌打出来。
- 这么做的目的不是单纯“贪大”，而是先把当前这手控制权稳住，再为下一拍把同门小牌递给同伴留出窗口。
- 这条规则同时承担一个很直白的信号作用：当我先把这门 `A` 兑现后，同伴就更容易把自己后续的 `A / K` 当成已经升高的控制牌来理解。
- 为了避免和旧递牌逻辑互相抢，只有当这门还留有后续小牌可递、且敌我双方都还没有公开绝这门时，才会触发这条“先出大牌”。
- 当初级没有明显的主控手，而公开信息又已经表明某位确定同伴在某门绝门时，会优先考虑打一张该门小牌，把牌权递给同伴。
- 这条规则和 `Pressure Void` 刚好相反：
- `Pressure Void` 是打敌方绝门，逼对手毙牌或暴露结构。
- `递牌` 是打同伴绝门，让同伴接手。
- 初级版递牌只看 `公开绝门`，不根据完整已出牌推断高张分布。
- 为了避免把接手权直接送给敌人，初级只在敌方公开上还没有显示也绝这门时，才会触发这条递牌。
- 这条规则同时覆盖“朋友回给打家绝门”的常见套路；如果朋友已明且打家公开绝门，初级会把这视为一种可执行的递牌窗口。

## 7. 跟牌

- 先满足同花、同张数、同型等规则约束。
- 如果自己已经是“确定朋友”，且当前该站队，就用最低成本的站队牌明身份。
- 如果庄家已经控制住这一轮，且站队可以安全延后，初级会先配合，不急着抢亮身份。
- 如果同阵营已经稳吃，初级更倾向喂分或为末局做简单准备，而不是无意义反超。
- 如果敌方当前领先且自己能以低成本压过，初级会抓明显能赢的机会。
- 如果自己已经缺首门、又不打算或不能用主成型去毙，初级不会因为别门刚好有对子或连对，就优先拿这些副牌结构去贴同型。
- 这时更合理的默认动作是：优先贴散副，尽量把副牌对子留作后续回手、续牌或抢末手资源。
- 如果当前是“缺首门跟单张”，而自己手里有 `10` 及以上、级牌或王组成的大对，同时还有一张 `5` 之类的低分散牌可贴，初级会优先贴这张低分牌，不轻易拆掉大对去抢一手单张控制。
- 但这条保大对短路不能盖掉仍然存在的安全毙牌；如果当前闲家正在拿分，而初级手里还有不拆这些高对也能压住的主牌，它会继续考虑毙牌，不会机械退回贴牌。

## 8. 末局

- 末手依然完全按正常出牌规则处理，不存在“因为只剩最后几张，所以允许乱打一把”的特权。
- 闲家在可见分偏低、剩牌偏少时，会进入轻量“抢扣底”倾向。
- 如果公开分数已经表明闲家把后面剩余分全拿也不够到 `120`，初级会更早切到“抢扣底 / 抢末手”思路。
- 如果公开分数已经表明闲家已经不可能靠常规分数翻盘，打家与朋友首发会更倾向先走低风险、低分值的牌。
- 打家和朋友在末局更倾向保底，不会为眼前小分拆掉关键保底牌。

## 9. 轻量级牌扣底 heuristic

- 当前版本已经落了一版轻量实现，但仍属于初级难度的克制 heuristic，不是完整末局搜索。
- 只要不是打家，玩家在开局就会先用自己的明手牌做一次“我有没有级牌扣底潜力”的轻量判断。
- 这次判断至少看三件事：
- 自己手里有没有真正能形成末手扣底的级牌，或者含有级牌的末手结构。
- 自己的主够不够长，是否有机会一路把关键主牌保到最后。
- 自己有没有一两张能在倒数第二手重新拿回牌权的大主，例如大王或足够稳的高主。
- 如果这三项同时较好，初级应当把“级牌扣底可能性”记成一个轻量局内目标，而不是只盯着常规跑分。
- 如果当前级别是 `J / Q / K / A` 这类特殊级，这个目标应当进一步升权。
- 因为这时即便只是把打家打降级，也等于显著推进了自己的长期升级进度。
- 如果此时自己还没有被叫到朋友，这个目标权重会适当提高；也就是说，闲家位会更主动把“争取级牌扣底”当成一条可能翻盘路线。
- 局内一旦朝这个方向倾斜，初级会更频繁地考虑吊主，先压缩打家保底和藏底的空间，再为末手留下窗口。
- 同时会更明确地保住大王、级牌，以及包含级牌的结构牌，尽量不要为了眼前小分过早拆掉这些末手资源。
- 如果后来被叫到了朋友，但并不是“叫死”的确定朋友，初级也不必立刻强站队。
- 这时更合理的思路是：在不明显违背同侧利益的前提下，保留一点“我是否马上当朋友”的犹豫，避免过早暴露身份，顺手把级牌扣底路线完全打碎。
- 当前这版仍有边界：
- 它只做“有无潜力”的轻量分层，不做完整末局路线搜索。
- “保大王 / 保级牌结构”目前主要体现在首发吊主与跟牌保留上，还没有扩展成统一评分器。
- 当前版已经同步开放给 `intermediate` 作为基础画像输入，但初级自身仍只保留轻量 heuristic，不做搜索。

## 10. 打家控局补强 heuristic

- 当前版本已经补了打家侧的第一版控局补强，目标是减少“朋友亮了但打家没切控局模式”和“朋友迟迟不亮还在机械找朋友”这两类输法。
- 如果朋友刚在前几轮亮相，打家和已经站队的朋友在接下来短窗口里会优先切到 `清主 > 续控 > 跑分 > 弱门试探`。
- 这条规则当前先落在首发层，意味着亮友后短时间内，AI 会优先出可消耗主、继续控主，尽量不立刻回到弱侧门试探。
- 如果当前是无主，且打家前几轮自己已经有足够明显的主控资源，例如双王、较长主或稳定主结构，那么初级会先打控制线，而不是急着先摸朋友门。
- 这条规则的目标不是永远不找朋友，而是先避免“还没站稳牌权，就把节奏和高张一起送回给闲家”。
- 如果牌局已经拖到第 `6` 轮左右朋友仍未亮，打家会切到一版 `solo banker fallback`。
- 这时它会降低继续找朋友门的频率，优先走主和低分安全牌，先把自己从“理想双人协同”切回“先保底、先保控轮”的模式。
- 当前这 3 条补强仍有边界：
- 它们主要体现在首发入口的优先级调整，还没有完全扩展到统一跟牌评分器。
- “亮友后控局”和“晚亮友 fallback”现在都属于轻量 heuristic，不是完整多轮搜索。
- 无主少探朋友门也只在前几轮、且打家已经明显有控制资源时生效，不会覆盖所有无主开局。

## 11. 和记牌更强难度的区别

- 初级不会因为“猜队友大概率缺门”就主动设计复杂回牌。
- 初级不会因为“打家已经出过这门部分高张，所以剩余高张大概率在同伴手里”而主动设计软递牌。
- 初级不会根据大量已出牌推断“某张关键牌已经绝了没”。
- 初级不会为了两轮后的结构收益，牺牲当前明显顺手的一轮。
- 中级与高级可以在此基础上继续做更复杂的“回手、续控、保结构、控末手”判断，但不应反过来污染初级定位。

## 12. 代码阅读导图

这一节不是新增规则，而是把当前实现里最容易混淆的入口和 helper 用“读代码语言”串起来，方便后续维护时快速定位。

### 12.1 初级总入口

- 初级 AI 的实际入口是 `src/shared/ai.js` 里的 `getLegalHintForPlayer`。
- 当难度是 `beginner` 时，它会直接路由到 `src/shared/ai-beginner.js` 的 `getBeginnerLegalHintForPlayer`。
- 这意味着初级不会经过中级的 objective 权重层，也不会走中级的 rollout 评分链。

### 12.2 `getBeginnerLegalHintForPlayer` 在做什么

- 这个函数是初级 AI 的总调度器。
- 如果当前是首发：
- 先检查末手合法首发、强制站队、特殊找朋友首发等直白规则。
- 如果这些都没命中，再按“推土机 / 火车 / 拖拉机 / 三张 / 对子 / 单张”的顺手顺序兜底。
- 如果当前是跟牌：
- 先枚举所有合法候选，再交给 `chooseAiFollowPlay` 选一手。
- 如果跟牌 heuristic 没给出明确结果，再按“尽量同门、尽量同型、实在不行保证合法”的顺序兜底。

### 12.3 `chooseAiLeadPlay` 在做什么

- `chooseAiLeadPlay` 只处理“首发时那几类值得单独拦截的场景”。
- 它的优先级大致是：
- 强制亮友
- 打家前几轮的找朋友铺路
- 无主打家保留朋友牌后的强势清大牌
- 需要主动亮友的首发
- 防止庄家将吃的安全首发
- 根据公开断门信息施压的首发
- 如果都不命中，就返回空，交给总入口做通用结构兜底。

### 11.4 `chooseAiFollowPlay` 在做什么

- `chooseAiFollowPlay` 只关心“已经拿到一批合法候选后，该选哪一手”。
- 它不会暴力搜索后续几轮，而是按当前局面做简单取舍：
- 如果必须或适合亮友，优先处理亮友。
- 如果当前不该亮友，但可以先暗中配合打家，就优先配合。
- 如果敌方当前领先且我能低成本压过，就从能赢的候选里选“结构更对、牌力刚够、送分更少”的。
- 如果我方已经稳吃，就优先不反超，转而喂分或为末局做简单准备。
- 如果进入抢扣底阶段，就会优先选择不会反超、但更利于末局垫牌的组合。

### 11.5 `getFollowStructureScore` 在做什么

- 这个函数是初级跟牌排序器的核心。
- 它最看重三件事：
- 有没有尽量跟对花色。
- 有没有尽量完整跟对牌型。
- 在对子、三张、拖拉机、火车、推土机这些结构场景里，有没有保住“必须出的结构单位”。
- 可以把它理解成：它不一定选“未来最好”的牌，但会尽量选“这轮看起来最像一个合格人类会出的合法牌”。

### 11.6 初级大量行为其实来自共享 helper

- `src/shared/ai-beginner.js` 里很多函数名像是“初级 AI 自己在做决定”，但实际身份协同、亮友、延迟亮友、防将吃、末局喂分等逻辑，很多落在 `src/shared/ai-shared.js`。
- 维护时如果只看 `ai-beginner.js`，容易误以为某些行为不存在；实际上只是被下沉到了共享 helper。
- 最常见的几个 helper 包括：
- `getForcedCertainFriendRevealPlay`：确定成友后强制亮身份。
- `chooseAiSupportBeforeReveal`：该延迟亮友时，先用低成本跟牌暗中配合。
- `chooseAiSafeAntiRuffLead`：避免给庄家留下将吃窗口。
- `chooseAiHandoffLead`：利用公开绝门信息把牌权递给已知同伴。
- `chooseAiVoidPressureLead`：利用公开断门信息去施压。
- `chooseAiBottomPrepDiscard`：残局抢扣底时优先垫哪类牌。

## 12. 当前实现中的关键落点

- 叫朋友与扣底联动逻辑主要落在 `src/shared/game.js`
- 初级出牌入口主要落在 `src/shared/ai-beginner.js`
- 站队、延迟站队、防将吃、末局喂分等共享 heuristic 主要落在 `src/shared/ai-shared.js`

- 如果是 beginner 实际行为变化，至少同步更新：
