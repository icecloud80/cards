这份文档把当前 Web 原型扩展为 `Mobile / H5 / 小程序 / PC desktop` 多端产品线的目标、功能范围、体验要求与商业化边界统一整理下来，作为后续实现、设计、测试和运营的共同参照。

## 1. 文档目标

- 明确短期 App 首发不是“重做一套新游戏”，而是把现有网页原型产品化、原生化、可运营化。
- 明确免费运营与广告变现的边界，避免首发阶段为了短期变现伤害留存。
- 明确多人在线是后续重点方向，但不进入首发强依赖。
- 明确当前 `HTML` 单机版将长期保留，但未来正式多端客户端主线会切换到 `Cocos Creator`。
- 为移动端 UI、数据、账号、广告、运营、QA 提供统一口径。

## 2. 当前项目现状

- 当前仓库已经具备 `PC / mobile` 两套平台壳层。
- 核心玩法、AI、日志和大部分状态机都在共享层实现。
- 当前最成熟的模式是 `1 名人类 + 4 名 AI` 的单机对局。
- 当前已有 headless 与浏览器 UI smoke 回归，可作为 App 化前的基础质量门槛。
- 当前进度主要保存在浏览器 `cookie / localStorage` 中，尚未具备账号与云存档。

这意味着 App 版最合理的首发策略是：

- 先把单机模式做成稳定的手机应用。
- 再把共享规则、AI、回放与状态机抽成可复用核心。
- 然后以 `Cocos Creator` 建立正式主线客户端，逐步覆盖 `H5 / iOS / Android / 小程序 / PC desktop`。
- 最后再在正式主线客户端上开放好友房、快速匹配和排位。

### 2.1 已确认的产品线

- `HTML Legacy`
  - 长期保留的无服务器单机版。
  - 主要承担离线游玩、规则验证、QA、回放与 AI 调试。
- `HTML / Capacitor` 过渡线
  - 当前阶段的可发布手机 App。
  - 继续承接单机首发、真机体验验证与基础商业化验证。
- `Cocos Mainline`
  - 未来唯一正式客户端主线。
  - 后续正式 `H5 / iOS / Android / 小程序 / PC desktop` 都优先从这条线输出。
- `Referee Server`
  - 未来正式联机服务端。
  - 不参与 `HTML Legacy` 离线单机版的启动和运行。

## 3. 产品定位

### 3.1 核心定位

- 一款面向熟人牌局与 AI 练习的 `找朋友升级` 手机游戏。
- 首发以 `单机 AI 对战 + 规则清晰 + 上手顺畅 + 复盘方便` 为核心卖点。
- 后续以 `好友房 + AI 补位 + 轻社交` 为多人方向。
- 产品表达上要明确强调：玩家的长期目标不是只赢一局，而是持续升级通关，先经过 `A 级`、`王级`，再进入更高轮次的 `+2 / +A` 级别。
- 如果玩家已经打到 `A`，后续再因级牌扣底降级时，产品文案必须明确展示完整负级链：`-K -> -Q -> -J -> -10 ... -> -2`，且 `-2` 为最低档。
- 这条负级链及其恢复顺序 `-2 -> -3 -> ... -> -K -> -A -> 2` 需要作为正式对外规则完整公开，不能只藏在实现、测试或结果页角落里。

### 3.2 用户价值

- 老玩家可以随时开一局练牌、练规则、练叫朋友与扣底思路。
- 新玩家可以借助规则文档、清晰 UI、托管与复盘更快入门。
- 没有固定牌搭子时，也能稳定游玩，不被凑人门槛阻断。

### 3.3 商业定位

- 免费下载、免费游玩。
- 以 `局间广告 + 激励视频 + 轻外观付费可能性` 为主，不做破坏公平性的数值售卖。
- 广告收入要建立在留存和时长健康的前提上，不牺牲核心对局体验。

## 4. 首发版本目标

首发版本目标统一定义为：

- 支持 `iOS` 与 `Android` 真机稳定运行。
- 支持单机完整 5 人对局，含当前三档 AI。
- 支持本地进度保存、继续游戏、基础设置同步。
- 首发这轮 App 存储迁移只要求原生壳稳定保存 `牌面 / AI难度 / 对局节奏 / 托管模式里的跨局态 / 玩家等级进度 / 最近一局复盘输入`；
  其中“最近一局复盘输入”只包含 `回放种子 + 开局码`，用于问题复盘与手动重建开局，不承诺中途续打恢复。
- 当前复盘输入统一使用更短的字母数字混合短码：`回放种子` 固定 `11` 位，`开局码` 固定 `169` 位，且开局码区分大小写。
- 支持新手查看规则、开始页选难度、选节奏、切牌面。
- 牌面资源需同时支持经典单张 SVG 与整图 sprite；当前 mobile 默认仍使用 `poker.png` 的 `经典整图`，并新增 `m_cards_sprite.png` 对应的 `现代整图` 主题供玩家切换。
- `m_cards_sprite.png` 不再只是仓库实验资源，而是正式接入 mobile 与 PC 的运行态牌面列表；两套整图主题都必须共用同一套 13x5 裁切规则。
- mobile 小卡位、顶部状态牌、出牌区与翻底牌背在任一整图主题下都必须复用同一张 sprite 资源，避免不同区域继续出现切图口径不一致的问题。
- 旧存档中的 `modern-sprite` 仍需继续可用；如果当前平台显式提供该主题，应直接进入 `现代整图`，避免用户升级后牌面设置失效。
- 手游手牌区的花色标签默认只显示花色文字，不显示张数；张数统计属于桌面端读牌辅助信息，不应继续挤占 mobile 的横向空间。
- 当前 `index2.html` 与 `index-app.html` 的移动端手牌区高度口径也要继续统一：在已有五行手牌布局上，当前已在上一版基础上累计再增加 `20px` 缓冲，确保真机窄屏下不会把分组牌面压得过紧。
- 当前 `index2.html` 与 `index-app.html` 的移动端底部操作区重新收口为同一套高度口径：两端都沿用相同的轨道高度、常规态高度与候选态高度，避免浏览器运行页和 App 壳继续形成两套拇指操作记忆。
- 当前 `index2.html` 与 `index-app.html` 的移动端底部操作按钮也必须在操作区内沿垂直中线居中，按钮文字本体同样要保持居中，避免同样的高度口径下仍出现“视觉上贴顶或贴底”的错觉。
- 当前 `index-app.html` 的手牌区又在现版基础上再抬高 `10px`，累计已比最初版本额外增加 `30px`；同时 App 牌桌容器底部 padding 继续减少 `5px`，把新增空间优先还给手牌区，而 setup 态主入口则只保留在开始页壳层，不再塞回局内底部操作条。
- 当前 `index1.html`、`index2.html` 与 `index-app.html` 的局内可见主操作区统一收口为 `选择 / 出牌` 两枚高频按钮；旧的 `亮主 / 不反 / 不亮 / 毙牌 / 新的游戏 / 继续游戏 / 开始发牌` 都不再保留为底部常驻按钮节点。
- 发牌亮主、补亮与反主的人类入口统一收口到共享 `setupOptions` 区：无人亮主时显示“所有可亮方案 + 不亮”，别人已亮主后切成“所有可反方案 + 不反”，最终反主读秒里点 `不反` 要能立刻结束读秒继续流程。
- 别人扣牌时，底部共享 setup 区不再显示任何可点动作，只直接显示 `玩家扣牌中` 状态文案；移动端按钮放大需求只继续作用于 `选择 / 出牌` 这两枚高频主按钮。
- 手游顶部 `主 / 朋` 状态牌也必须复用当前牌面主题，不允许一部分状态卡走 sprite、另一部分退回单张 `img`；同一条顶部状态带里的牌宽、留白和阴影口径必须一致。
- `index2.html` 与 `index-app.html` 顶部 `主 / 朋` 状态牌即使处于“可重选朋友牌”的点击态，也应继续保持纯牌面展示；不再额外叠加橘色描边或卡框，避免误读成另一层状态优先级。
- 为了支持移动端评审、截图和离线视觉对齐，仓库需要额外维护一个独立于共享运行时的 `index2-static.html` 手游静态牌桌页；该页面应固定提供一套 mock 手牌、上一轮、底牌和菜单交互样例。
- 支持基础埋点、崩溃监控、公告、远程配置。
- 支持轻量广告变现，但广告不进入牌桌主战场。
- 这轮首发客户端仍属于 `HTML / Capacitor` 过渡版本，不代表未来正式多端主线继续沿用同一套渲染壳层。

首发版本明确不包含：

- 实时多人匹配。
- 排位、赛季、排行榜。
- 公会、聊天、送礼。
- 会影响公平性的付费增益。

## 5. 用户分层

### 5.1 新手用户

- 特征：会玩扑克牌，但不熟这套五人找朋友规则。
- 诉求：看得懂阶段、知道何时亮主、何时扣底、何时叫朋友。
- 重点能力：新手规则入口、关键播报、托管、推荐出牌提示、复盘说明。
- 托管入口需保持三态且一键循环：`关闭 / 本局托管 / 跨局托管`，避免 mobile 与 PC 的产品口径分裂。

### 5.2 熟练玩家

- 特征：懂规则，想练 AI、练叫朋友路线、练牌权续控。
- 诉求：更快进入对局、少打扰、AI 有层次、日志可复盘。
- 重点能力：快速开局、节奏档、日志导出、结果摘要、AI 难度区分。
- 这类用户还会期待 AI 能体现“特殊级里级牌扣底更值钱、未站队阶段会提前为级牌扣底布局”的真实策略差异。
- 同时也会期待共享 AI 遵守熟手牌感：比如缺首门贴副时，不会把副牌对子当成“顺手垫牌”轻易送掉。
- 同时也要避免“仍有首门时为省零分乱拆同门结构”的反直觉走法；例如延迟站队跟单张时，若手里是 `8899 + 5`，应优先跟 `5` 保住 `8899`，而不是随手拆拖拉机。

### 5.3 社交牌友

- 特征：未来更看重好友房、固定队、熟人局。
- 诉求：建房方便、掉线可回、凑不齐也能 AI 补位。
- 重点能力：邀请码建房、好友列表、最近同桌、回放与举报。

## 6. 核心玩法范围

App 版首发玩法与当前 Web 原型保持一致：

- 5 人对局。
- 3 副牌。
- 亮主 / 反主 / 翻底定主。
- 扣底 / 保底 / 级牌扣底。
- 打家叫朋友与第几张判定。
- 三档 AI 与节奏档位。
- 在 `J / Q / K / A` 这些特殊级里，产品说明与 AI 表现都应明确体现：级牌扣底不是普通附带项，而是高优先级整局目标。
- `A` 级被继续降级时，必须按 `-K -> -Q -> -J -> -10 ... -> -2` 的完整顺序执行；结果页、存档和复盘都要沿用同一口径。

产品化过程中不应为了适配 App 而改动这些规则。

如果未来为了联网、防作弊或新手体验引入额外限制，应优先更新规则文档，再同步实现。

## 7. 功能范围

### 7.1 首发必须有

- 单机开局与继续游戏。
- 首页 `继续游戏` 在检测到可继续进度后，必须直接恢复当前等级并进入发牌，不能只刷新按钮态或停留在首页。
- `index2.html` 与 `index-app.html` 的开始页继续按钮状态必须直接读取共享 render 快照里的 `hasSavedProgress`；共享层若漏掉这项状态，视为阻断性回归。
- 局内 `刷新` 入口。
- mobile 局内 `刷新` 只能保留顶部高频入口，设置菜单不再重复提供同义按钮。
- AI 难度切换。
- 节奏切换。
- 托管。
- 托管必须支持 `关闭 / 本局托管 / 跨局托管` 三态；其中“跨局托管”只改变托管生命周期，不改变 AI 难度与规则边界。
- `刷新` 只允许按当前 `回放种子` 重开本局，必须保留玩家当前级别，不能把长期进度清回 `2`。
- `刷新` 必须固定使用顶部 `icon-refresh.png` 图标资源，并保持在 `上一轮回看` 与 `设置` 之间。
- 规则查看。
- 首页 `规则` 与局内 `规则帮助` 都必须直接打开共享规则面板，不能再通过点击旧隐藏按钮或“更多设置”入口间接代理。
- 局内 `规则帮助` 文案必须改成更像“边玩边看说明”的短句版，优先覆盖 `游戏目的 / 胜利条件 / 升级分数要求 / 亮主 / 翻底定主 / 扣底 / 叫朋友 / 跟牌 / 末手 / JQ 级主副级牌扣底 / 负级简写` 这些高频疑问。
- 结果页与等级变化展示。
- 对局日志查看与导出。
- 局内信息栏只展示适合运行时扫读的短头部信息，不直接展示 `回放种子 / 开局码` 这类复盘长文本。
- 日志导出至少要包含 `首抓玩家 / 玩家等级 / 回放种子 / 开局码`，保证客服、QA 和 AI 复盘都能仅凭日志还原开局输入；其中当前短码口径固定为 `11 + 169` 位字母数字混合文本。
- 若当前打开 AI debug，日志导出还要补齐 `亮主 / 反主 / 扣底 / 叫朋友 / 首发 / 跟牌` 这些 AI 决策阶段，避免手机问题局导出后只剩半截策略链。
- 设置菜单需要补入 `复制复盘码`：把当前局 `回放种子 + 开局码` 一次性复制成紧凑串，方便从手机直接发到 PC 或 QA 群继续复盘；其中 `开局码` 需保留原始大小写。
- 牌面切换。
- 本地进度保存。
- 玩家等级进度、本地存档、结果页和开局码元信息必须完整支持 `Lv:-K` 到 `Lv:-2` 的负级展示与回放，不得再写死成旧的 `-A / -2` 两档。
- 基础埋点与崩溃上报。
- 公告与远程配置。
- 局间广告与激励视频能力。

### 7.2 首发建议有

- 新手引导。
- 局内设置页。
- 音效与震动开关。
- 横竖屏策略统一。
- 弱网与切后台恢复。
- 低电量 / 低性能机型降级策略。

### 7.3 首发可延后

- 账号系统。
- 云存档。
- 好友系统。
- 皮肤商城。
- 联机房间。
- `Cocos Mainline` 正式版客户端。
- 小程序版与桌面版正式输出。

## 8. App 端专属需求

### 8.1 设备与系统

- `iOS`：近 3-4 个大版本内主流系统。
- `Android`：覆盖中端与主流国产机型，优先保证中低端机流畅。
- 需要适配刘海屏、挖孔屏、全面屏安全区。
- 需要处理横竖屏、来电、切后台、锁屏恢复等系统事件。

### 8.2 本地能力

- 应用图标、闪屏、权限说明。
- 应用图标首版必须摆脱默认壳占位风格，统一采用 `亮红背景 + 叠牌主体 + 红桃A牌面` 的品牌口径；iOS 1024 图标、Android legacy icon 与 adaptive foreground/background 必须同时成套交付，不允许只换其中一端。
- 本地持久化存档。
- 版本更新提示。
- 首发这轮只要求“最近一局复盘输入”可跨重启保留；崩溃恢复后继续未完成对局属于后续能力，不在本轮落地。
- 音效、震动、屏幕常亮策略。

### 8.4 首发 App 壳初始化口径

- 首发 App 名称固定为 `找朋友升级`。
- 首发包名固定为 `com.nolanli.cards`。
- 首发 `Capacitor` Web 输出目录固定为 `dist/app`，作为 iOS / Android 原生壳同步入口。
- `dist/app/index.html` 首阶段直接复用当前 `index-app.html` 生成，确保原生壳默认进入 App 专用手游运行态，而不是继续落到仓库预览或通用 `index2.html`。
- `index-app.html` 需要和 `index2.html` 做入口隔离：前者专门服务 iOS / Android 原生壳，后者继续保留给浏览器预览、对照和 smoke 回归。
- App Web 构建只允许携带当前运行实际需要的 HTML、`src/` 与 `images/` 资源；像 `elements.cardmeister.min.js` 这类已废弃脚本必须从构建清单和原生壳同步产物里移除。
- iOS 原生工程的本地开发状态文件必须继续和业务资源隔离管理；`*.xcuserstate`、`xcuserdata`、`DerivedData` 这类仅对当前机器生效的 Xcode 产物不得进入版本库，避免不同开发机之间互相污染工程状态。
- 如果历史上已经把 `.DS_Store` 或 Xcode 本地状态文件提交进仓库，后续治理需要显式清理 Git 索引；单纯补忽略规则只影响未来新增文件，不会自动回收历史脏数据。
- 这次初始化只确定 App 壳元数据与打包入口，不改变任何玩法规则、AI 难度、广告边界和版本范围。

### 8.3 运营能力

- 埋点事件。
- 远程配置。
- 公告弹窗。
- 广告位配置。
- 商店评分引导。
- 客服与反馈入口。

## 9. UI / UX 需求

### 9.1 移动端 UI 原则

- 以“当前局面优先，规则解释次之”为原则。
- 牌桌主战场必须始终让玩家看清：
  - 当前主牌。
  - 当前阶段。
  - 当前轮到谁。
  - 闲家总分。
  - 朋友状态。
- 所有强操作按钮都要满足单手点击与误触可撤销。
- 重点弹层不能遮住关键牌面太久。
- 所有首发交互都应能在 6.1 至 6.7 英寸手机上舒适完成。

### 9.2 移动端重点改进项

- 开始页改成真正的 App 首页，而不是网页原型入口拼接。
- 手游运行页、静态模板页和 App 壳页的品牌标题统一使用 `找朋友升级`；开始页标题、局内标题与页签不再混用 `五人找朋友升级`。
- App 首页、商店素材与桌面图标应共享同一套品牌语言：延续 `亮红底 + 叠牌元素 + 红桃A`，避免主屏图标、首页首屏与提审截图继续像三个不同产品。
- 开始页和设置页里的 `对局节奏` 都要使用和 PC 相同的四档按钮组，直接展示 `慢 / 中 / 快 / 瞬`；mobile 不应再把节奏主入口做成原生下拉框。
- 开始页和设置页里的 `对局节奏` 标题行不再额外挂“与 PC 一致”提示字样；跨端一致性由按钮形态和档位语义本身表达，避免在窄屏上重复占用说明位。
- 局内顶部状态栏进一步收紧，减少文字解释，保留短标签与可视徽章。
- App 专用牌桌必须改成 `顶部状态固定高度 + 中部出牌区自适应 + 底部手牌固定高度 + 底部操作固定高度`；不能再把顶部状态区、出牌区和手牌区都写死在同一组行高里。
- App 专用页面的安全区处理要按原生 WebView 收口：顶部只保留必要的刘海安全边，底部只保留轻量手势条预留，不再沿用移动浏览器底栏那套大留白。
- App 底部操作条必须压成单行薄轨：按钮区不能再次叠加 `safe-area-inset-bottom` 内边距，验收口径以“最后一行手牌完整可见、不被主操作条上顶遮挡”为准。
- App 底部操作条的按钮列数要随当前可见按钮自适应均分，且 `选择 / 出牌` 这类主按钮下方不允许残留被手机高 DPR 放大的空白轨道。
- App 手牌区需要略大于当前版本，并把手牌牌面直接对齐到中央出牌区同一套尺寸；不能继续出现“中央看得清、底部手牌偏小”的双口径。
- App 与 `index2.html` 运行页在无主场景下都要稳定容纳 `主牌 + 梅花 / 方块 / 黑桃 / 红桃` 共 5 行；不能再把某一门花色横向挤到主牌右侧，所有花色标签左边必须保持同一条基线。
- `index2.html` 需要把“常规出牌态”和“亮主/反主态”分开处理：`选择 / 出牌` 这类单行主按钮可放大到约 `1.5x` 并保留明显上下 padding，但 `亮主 / 反主` 的候选区必须继续保持紧凑，不能因为底部主按钮变厚而被挤到屏幕下方。
- `index2.html` 的手牌轨不仅要对齐花色标签，还要保证 `主牌` 行与下面各花色行的第一张牌起点一致，避免大王那一行看起来比下面花色牌更靠左或更靠右。
- 当顶栏高频按钮数量调整时，`主 / 朋` 这类中部状态块需要优先避免压住 `计秒 / 总分 / 难度` 等左侧统计；必要时应整体右移，而不是继续压缩统计可读性。
- 顶部 `主 / 朋` 状态卡的点击反馈应优先交给文案、角标与点击行为本身，不再通过额外橘色描边强调可点击态，避免主牌和朋友牌看起来像被多包了一层边框。
- 手牌区强化选中反馈、误触撤销与长按预览。
- 叫朋友、亮主、反主、扣底等关键阶段改成更明确的步骤化面板。
- 设置页需要从浮层式网页菜单升级为 App 风格设置页。
- 手游静态牌桌模板页也要继续复用同一套设定菜单形态，至少保留 `对局节奏 / 返回首页 / 牌面 / 底牌 / 规则帮助`，避免评审稿和运行态菜单结构脱节。
- 顶栏高频按钮需固定包含 `上一轮回看 / 刷新 / 设置` 这组连续入口；`刷新` 必须统一使用 `icon-refresh.png`，并保持在 `上一轮回看` 与 `设置` 之间。
- 这条 `刷新` 入口固定复用当前局 `replaySeed`，作为“重开当前随机链路”的高频快捷方式。
- `上一轮回看` 面板需优先采用紧凑横排行：每位玩家左侧只保留玩家名与身份摘要，右侧横向展示上一手牌列，避免 5 位玩家纵向堆叠把浮层整体拉得过高。
- 设置页只保留 `返回首页 / 牌面 / 布局 / 底牌 / 规则` 这类低频项，不再重复承载顶部已经直出的 `刷新` 高频操作。
- 设置页现已额外保留 `复制复盘码` 这一条跨端调试入口；它属于低频工具，但对手机问题局复现非常关键，应和 `牌面 / 底牌 / 规则` 同层收纳。
- 这条入口当前复制出的内容必须统一为 `11` 位回放种子 + `169` 位开局码的短码组合，避免手机端继续生成旧的超长 hex 文本。
- 结果页里的 `复盘日志` 需要优先承接完整 AI 策略链，尤其要能看到 `扣底 / 叫朋友` 这类过去容易缺失的中间阶段。
- 结果页增加“再来一局 / 复盘日志 / 返回首页”清晰分流。
- 手游结果页在 6.1-6.7 英寸设备上必须优先保证首屏读完：标题、正文、级别结算、底牌亮出和四枚操作按钮都要尽量压缩留白，必要时只允许卡片内部轻量滚动，不能默认整张弹窗超高撑屏。

### 9.3 平板与大屏策略

- 平板不是简单放大手机版。
- 需要在保持移动操作逻辑的前提下，适度恢复更多局面信息常驻展示。
- 大屏优先让对局桌面更完整，而不是把所有面板都做得更大。

### 9.4 PC 与移动体验关系

- 当前仓库已有 PC 独立布局方向，后续继续保留。
- 当前三端主战场母稿已统一补到 [three-surface-ui-mock.md](three-surface-ui-mock.md)；后续若继续推进 `PC / H5 / App` 的同轮 UI 精修，应优先以这组设计稿校准信息层级、操作热区和视觉语言，再分别落到各端模板。
- PC ready 阶段的开始界面主视觉应与手游开始页共享同一套插画元素，避免两端首页入口继续分叉成两套不同品牌画面。
- PC ready 阶段的开始界面标题应直接统一为 `找朋友升级`，不再额外叠加 `PC 开始界面` 这类平台提示标签，避免品牌名和场景名同时出现造成首屏噪音。
- PC 局内顶部状态条左侧统计也应与手游尽量对齐为 `计时 / 总分 / 难度` 这类短信息组合，减少玩家跨端切换时的理解成本。
- App 化阶段不要求 PC 与 mobile 完全统一视觉，但需要统一核心术语、状态块与结果表达。
- `AI难度`、`对局节奏` 这类开局级设置在跨端上应优先统一操作语义；当 PC 已升级为按钮组时，mobile 端也要同步成相同形态，避免一端按钮、一端下拉的割裂体验。
- 后续如要做跨端账号与联机，需要先统一阶段名称、提示文案、操作语义与日志结构。

## 10. 商业化需求

### 10.1 总原则

- 广告不能打断关键操作。
- 广告不能出现在玩家思考和出牌主流程中。
- 广告不能绑定胜负收益。
- 激励奖励只能给非公平性收益。

### 10.2 首发推荐广告位

- `局间插屏`：结算页关闭后、下一局开始前。
- `激励视频`：用于领取头像、牌背、桌布、额外复盘信息、免广告时段等奖励。
- `首页原生位`：可选，仅在后续留存稳定后考虑。

### 10.3 首发明确不推荐

- 对局中底部 Banner 常驻。
- 关键按钮旁诱导点击广告。
- 与等级、胜率、AI 强度直接绑定的付费或看广告收益。

### 10.4 广告奖励边界

- 可给：
  - 外观资源。
  - 额外复盘页。
  - 当日双倍任务进度。
  - 短时免广告。
- 不可给：
  - 更强手牌。
  - 额外撤销次数影响胜负。
  - 付费直升段位。
  - 联机对局中的公平性增益。

## 11. 账号与数据需求

### 11.1 首发阶段

- 支持游客身份。
- 本地保存玩家等级进度、设置、牌面偏好、日志偏好。
- 不强制登录。

### 11.2 第二阶段

- 支持 Apple 登录与 Google 登录。
- 支持游客绑定。
- 支持云存档同步。
- 支持多设备恢复最近进度。

### 11.3 后续联机阶段

- 账号 ID 成为好友、房间、举报、战绩和广告分群的统一主键。

## 12. 数据指标

### 12.1 首发核心指标

- 安装转化率。
- 新手完成首局率。
- 首日留存。
- 7 日留存。
- 人均单日局数。
- 人均单日时长。
- 广告展示率。
- 激励视频完成率。
- 崩溃率。

### 12.2 联机前置指标

- 完成单机 3 局以上用户占比。
- 使用过日志 / 复盘的用户占比。

- 不同 AI 难度的选择分布。

这些指标将决定多人在线是否已经具备上线价值。

## 13. 风险与质量门槛

### 13.1 主要风险

- 直接网页封装后，真机性能和内存表现不稳定。
- 广告接入后打断原有节奏，导致留存下滑。
- 存档仍依赖浏览器式逻辑，切后台或升级后丢进度。
- AI 在真机长时对局中暴露新的卡回合或渲染问题。
- `HTML / Capacitor` 过渡线与 `Cocos Mainline` 双线并行后，需求范围若不及时收敛，容易反复返工。
- 如果共享 `game-core` 抽离不彻底，后续 `Cocos`、联机服务端和旧 HTML 版会继续互相牵制。

### 13.2 质量门槛

- 首发前必须完成真机长局稳定性测试。
- 关键流程必须覆盖：
  - 新开一局。
  - 继续游戏。
  - 亮主 / 反主。
  - 扣底 / 保底。
  - 结果页跳转。
  - 广告展示失败回退。
  - 切后台恢复。
- 每个版本都要保留回归清单与 bug 清单。

## 16. 2026-03-16 局内声明交互补充

- 声明候选区里的“跳过当前动作”需要显式按钮表达，不能让用户只能靠等倒计时或猜测是否可以不操作。
- 当进入“其他玩家都没亮主，等待玩家1补亮”的 15 秒窗口时，界面必须提供明确的 `不亮` 按钮；点击后应立即进入翻底定主，不再继续等待倒计时。
- 最后反主阶段的跳过按钮文案固定为 `不反`，不能出现空白、键名或 `undefined` 这类技术占位。
- 这次补充只调整局内声明交互表达，不改变广告、商业化、联网边界和版本拆分。

## 17. 2026-03-16 自动选择性能补充

- 局内 `自动选择 / 提示` 属于高频交互，主牌拖拉机这类结构牌型必须保持近实时反馈，不能因为候选层回落到大规模组合枚举而出现明显卡顿。
- 这条性能要求属于共享玩法体验约束，PC 与 mobile 都要遵守；即使问题先在 PC 暴露，移动端也不能接受同类退化。
