# 五人找朋友 App 与多人在线技术设计

这份文档给出当前 Web 原型走向 `iOS / Android App` 以及未来支持多人在线时的推荐技术架构，重点解决 4 个问题：

1. 如何最低风险地把当前项目变成可发布的手机 App。
2. 如何在不推倒现有共享规则层的前提下支持账号、云存档和广告。
3. 为什么多人在线必须采用服务端权威判定。
4. 好友房、快速匹配、AI 补位与回放分别应该落在哪一层。

配套产品目标见 [mobile-app-product-requirements.md](mobile-app-product-requirements.md)。

补充阅读：

- 首发任务拆解见 [mobile-app-launch-task-breakdown.md](mobile-app-launch-task-breakdown.md)。
- 联机接口与状态机草案见 [mobile-online-api-state-machine.md](mobile-online-api-state-machine.md)。

## 1. 设计原则

- 优先复用当前共享规则、AI 与状态机，不重写玩法核心。
- 客户端负责渲染、交互和本地体验；服务端负责联机状态、合法性校验与风控。
- 单机与联机尽量共用同一套操作语义、日志结构和结果结构。
- 先保证单机 App 稳定，再逐步引入账号、云存档和联机能力。
- 联机阶段必须避免客户端互信，不能把“是否合法出牌”交给本地自行裁定。

## 2. 当前代码基础判断

当前仓库已经具备 3 个对 App 化和联机都很有价值的基础：

- `src/shared/` 中已经沉淀了共享玩法、AI 与 UI 辅助逻辑。
- `src/platform/mobile.js` 已经说明项目存在移动端平台隔离入口。
- `tests/support/headless-game-context.js` 已经能在 Node 环境加载共享逻辑做同步回归。

这意味着：

- App 首发阶段可以直接采用 `WebView 容器 + 原生桥接`。
- 联机阶段可以尝试把共享规则裁剪后运行在服务端。
- 现有 headless 测试可演进为联机裁判层的回归测试基础。

## 3. App 端推荐架构

### 3.1 容器方案

推荐方案：`Capacitor + 当前 Web 前端`

原因：

- 当前项目是原生 HTML / CSS / JS 结构，迁移成本最低。
- 可以快速获得 iOS / Android 原生壳、插件和打包能力。
- 后续接本地存储、崩溃监控、广告 SDK、推送、商店能力都更直接。

不推荐首发直接重写成：

- Flutter。
- Unity。
- React Native。

原因不是这些框架不能做，而是当前项目已经有较成熟的共享前端和测试基础，重写成本和验证周期都更高。

### 3.1.1 当前工程初始化约定

- 首发 App 名称固定为 `找朋友升级`。
- 首发 App ID 固定为 `com.nolanli.cards`。
- `Capacitor` 配置文件固定放在仓库根目录，由 CLI 统一读取。
- `Capacitor` 的 `webDir` 固定为 `dist/app`，由 `npm run build:app-web` 生成。
- `dist/app/index.html` 首阶段直接由 `index2.html` 复制生成，让原生壳默认进入当前 mobile 运行态。
- `index1.html / index2.html / index-static.html` 仍随构建一起复制，便于后续原生壳调试、对照与 smoke 验证。
- Android 首阶段可直接通过 `npx cap sync android` 迭代；iOS 侧除了 `CocoaPods` 外，还需要本机安装完整 Xcode，并把 `xcode-select` 指向 Xcode.app，而不是只停留在 Command Line Tools。

### 3.2 客户端分层

推荐把客户端拆成下面几层：

#### 表现层

- 页面布局。
- 动画与过渡。
- 牌面渲染。
- 经典单张 SVG 与整图 sprite 的统一接入；当前 App 壳默认整图牌面应继续沿用 `poker.png`，并保留对 `m_cards_sprite.svg` 这类实验资源的接入能力，等重新通过视觉验收后再考虑切换默认入口。
- 弹层与提示。

#### 交互层

- 点击、拖动、长按。
- 手牌选中。
- 亮主 / 反主 / 叫朋友 / 扣底面板。
- 设置与规则入口。

#### 游戏应用层

- 开局。
- 继续游戏。
- 对局配置。
- 日志导出。
- 单机与联机模式切换。

#### 共享规则层

- 状态机。
- 规则合法性。
- AI 决策。
- 结果结算。

#### 原生桥接层

- 持久化。
- 广告。
- 埋点。
- 崩溃监控。
- 推送。
- 商店评价。
- `Capacitor` 插件桥接与原生壳元数据管理。

## 4. 首发 App 需要新增的原生能力

### 4.1 存储

- 把当前 `cookie / localStorage` 持久化逻辑迁移到 App 可控存储。
- 建议拆成：
  - `Preferences`：设置、牌面、节奏、是否看过引导。
  - `Preferences` 里的节奏字段继续统一沿用 `slow / medium / fast / instant` 四档枚举，保证 mobile App、当前 Web mobile 壳层和 PC 入口都能共用同一份配置值，不因为 UI 形态从下拉改成按钮组就分叉数据结构。
  - `SQLite` 或本地数据库：长日志、对局回放、可恢复进度。

### 4.2 生命周期

- 进入后台时冻结对局。
- 恢复前台时校验当前状态并安全恢复。
- 崩溃重启后支持“恢复上次未完成对局”。

### 4.3 监控

- 崩溃监控。
- JS 运行时错误监控。
- 启动耗时。
- 长局卡顿与内存峰值统计。

### 4.4 广告桥接

- 插屏广告。
- 激励视频。
- 广告失败降级回退。
- 远程控制频率与冷却。

### 4.5 远程配置

- 广告开关与频率。
- 功能灰度。
- 新手引导开关。
- 运营公告内容。

## 5. 数据模型建议

### 5.1 本地玩家数据

- `player_profile`
  - 本地玩家名。
  - 引导完成状态。
  - 常用设置。

- `player_progress`
  - 各席位等级进度。
  - 最近一次继续游戏快照。
  - 最近使用的 AI 难度与节奏。

- `match_history`
  - 时间。
  - 平台。
  - AI 难度。
  - 结果摘要。
  - 日志引用。

### 5.2 账号阶段云数据

- `user`
  - userId。
  - 登录方式。
  - 设备绑定信息。

- `cloud_progress`
  - 最近一次同步版本。
  - 等级进度。
  - 解锁外观。
  - 设置偏好。

### 5.3 联机阶段房间数据

- `room`
  - roomId。
  - 创建者。
  - 当前状态。
  - 座位分配。
  - 是否允许 AI 补位。

- `match_session`
  - matchId。
  - 当前局面快照。
  - 当前行动者。
  - 倒计时。
  - 日志事件流。

- `player_session`
  - userId。
  - roomId。
  - 在线状态。
  - 重连 token。
  - 托管状态。

## 6. 多人在线的推荐总体架构

### 6.1 总体原则

多人在线必须使用 `服务端权威判定`。

原因：

- 这套游戏规则复杂，包含亮主、反主、同门跟牌、甩牌、扣底、叫朋友第几张等多层约束。
- 如果把合法性判定放在客户端，本地改包、断网回放、时序竞态都很容易出错。
- 未来如果接排位、举报和回放，没有服务端权威日志会很难追溯。

### 6.2 推荐服务拆分

#### API / Gateway

- 登录。
- 建房。
- 匹配。
- 房间列表。
- 战绩查询。

#### Realtime Gateway

- WebSocket 长连接。
- 房间广播。
- 心跳。
- 重连恢复。

#### Match Service

- 房间状态机。
- 座位管理。
- 开局准备。
- AI 补位调度。

#### Game Referee Service

- 接收玩家操作意图。
- 校验是否轮到该玩家。
- 用共享规则层判定是否合法。
- 推进状态机。
- 生成标准事件日志。
- 广播最新局面。

#### Social Service

- 好友。
- 邀请。
- 最近同桌。
- 举报。

#### Analytics / LiveOps

- 行为埋点。
- 广告表现。
- 活动配置。
- 用户分群。

## 7. 推荐联机技术方案

### 7.1 客户端通信

- 协议：`WebSocket`
- 事件格式：明确区分 `command / event / snapshot / ack / error`
- 客户端只上报“意图”，例如：
  - `declare_trump`
  - `counter_declare`
  - `complete_burying`
  - `confirm_friend_target`
  - `play_cards`
  - `toggle_auto_manage`

### 7.2 服务端语言

优先建议：

- `Node.js / TypeScript`

原因：

- 当前共享规则层就是 JS。
- 最容易把可复用的合法性判断逐步抽到服务端共用模块。
- 现有 headless 测试与脚本也更容易接入。

### 7.3 存储与基础设施

- `PostgreSQL`：账号、战绩、房间元数据、配置。
- `Redis`：房间在线状态、倒计时、快速广播缓存。
- 对象存储：回放与长日志归档。

### 7.4 服务部署

- 首阶段可先单区部署，减少复杂度。
- 并发上来后再拆：
  - API 层横向扩容。
  - 房间服务按房间 ID 分片。
  - 日志与分析异步落盘。

## 8. 单机与联机共用逻辑的拆分建议

### 8.1 可共用

- 规则判定。
- 牌型识别。
- 结算逻辑。
- 日志结构。
- AI 行为。

### 8.2 需要隔离

- DOM 与渲染逻辑。
- 浏览器存储访问。
- 倒计时与动画时序。
- 本地提示文案与原生桥接。

### 8.3 推荐抽象

- `core/`
  - 纯规则与状态转移。
- `client/`
  - 渲染与本地交互。
- `server/`
  - 房间、联机协议、裁判与回放。

长期目标是让 `core/` 能被：

- 单机 App。
- 联机服务端裁判。
- headless 回归。

三端共同复用。

## 9. 回放与日志架构

### 9.1 为什么要优先建设

- 这类牌局解释成本高，用户经常会问“为什么这手不合法”“为什么这里扣底成立”。
- 回放是客服、风控、AI 调优、玩家复盘的共同基础。

### 9.2 推荐日志结构

- `match_created`
- `seat_joined`
- `deal_started`
- `trump_declared`
- `counter_declared`
- `bottom_revealed`
- `bottom_buried`
- `friend_target_confirmed`
- `cards_played`
- `trick_resolved`
- `round_finished`
- `match_finished`

每条事件建议都保留：

- `eventId`
- `matchId`
- `actorId`
- `timestamp`
- `phase`
- `payload`
- `snapshotVersion`

### 9.3 回放展示能力

- 首阶段：文本日志与关键结果摘要。
- 第二阶段：逐手回放。
- 第三阶段：联机观战与举报回放。

## 10. 掉线、重连与托管

### 10.1 基本原则

- 联机阶段必须允许掉线重连。
- 重连失败或超时后，自动转托管。
- 托管行为沿用现有 AI 难度设定，但联机房间内需要统一托管规则。

### 10.2 重连所需能力

- 客户端保存最近 `matchId / userId / reconnectToken`。
- 服务端保存最近房间快照与待确认事件序号。
- 客户端重连后先拉取完整快照，再补事件差量。

### 10.3 托管边界

- 短暂断线可保持座位。
- 超时后自动 AI 托管。
- 恢复连接后允许从下一次本方行动重新接管。
- 单机 / 当前 App 壳层里的手动托管统一使用 `关闭 / 本局托管 / 跨局托管` 三态；这里只有 `跨局托管` 会跨局保留。
- 联机里的“超时转托管”应等价于“本局托管”，不能自动升级成跨局持续托管，避免把掉线惩罚错误地延续到下一局。

## 11. 反作弊与风控

### 11.1 必须防的类型

- 非法出牌。
- 非轮到本人时发操作。
- 重放旧包。
- 篡改本地手牌数据。
- 客户端加速或伪造倒计时。

### 11.2 推荐措施

- 服务端权威手牌与局面快照。
- 每次操作带上 `matchVersion / commandId`。
- 服务端检查命令是否过期、重复或越序。
- 异常局自动标记、保留回放。

## 12. 广告与隐私合规架构

### 12.1 广告接入原则

- 广告决策留在客户端展示层与运营配置层。
- 广告不能进入核心规则层。
- 广告事件要和对局日志、留存指标分开埋点。

### 12.2 合规要求

- iOS 侧要预留 ATT 与隐私权限流程。
- 广告 SDK 要支持 consent 流程与失败回退。
- 所有广告位都要可远程关闭。

### 12.3 配置项建议

- 插屏冷却时间。
- 每日插屏上限。
- 激励视频奖励类型。
- 新用户前 N 局免广告。
- 无广告时段开关。

### 14.1 M1

- 单机 App 打包上线测试。
- 原生存储与崩溃监控就位。
- 广告桥接就位。

### 14.2 M2

### 14.3 M3

- 好友房。
- AI 补位。

### 14.4 M4

- 快速匹配。
- 战绩与举报。
- 回放与观战。

## 15. 技术债与提前规避点

- 不要继续把新存档能力直接堆在 `cookie` 上。
- 不要把联机房间状态只存在客户端。
- 不要让广告 SDK 直接侵入出牌流程和核心状态机。
- 不要把 DOM 依赖继续混进未来服务端要复用的规则函数里。

后续所有 App 化和联机实现，都应围绕“共享规则层纯化 + 客户端原生化 + 服务端权威化”这 3 条主线推进。
