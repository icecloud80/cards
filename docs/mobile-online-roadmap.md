# 五人找朋友 Mobile、多端与联机路线图

这份路线图用于统一当前项目的移动端、多端客户端与联机演进顺序。  
从 `2026-03-17` 起，项目的移动与多端主线正式调整为：

- 短期继续迭代当前 `HTML / Capacitor` 版本，作为过渡期可发布手机 App。
- 长期正式客户端主线切换为 `Cocos Creator`。
- 长期保留一个无需服务器的 `HTML 单机版`，用于离线游玩、规则验证、QA 与 AI 调试。
- 联机版统一走 `服务端权威判定`，并优先服务 `Cocos H5 / iOS / Android`。

这份文档已经替换掉旧版本里“把当前 `Capacitor + WebView` 视为长期主线客户端”的假设；后续排期、资源投入与多端决策都应以本文为准。

## 1. 已确认的产品线

### 1.1 `HTML Legacy`

定位：

- 长期保留的 `HTML` 单机版。
- 可静态部署。
- 不依赖服务器即可完整游玩单机对局。
- 主要承担离线体验、规则验证、QA、回放输入验证和 AI 调试。

边界：

- 默认不承接未来正式联机功能。
- 默认不承接小游戏、桌面客户端和长线运营能力的首发落地。
- 只保留对核心玩法、AI、复盘和基础 UI 的稳定维护。

### 1.2 `HTML / Capacitor` 过渡 App

定位：

- 当前阶段的可发布 `iOS / Android` 版本。
- 以“尽快验证手机端留存、手感与基本商业化”为目标。
- 继续复用当前 `HTML / CSS / JS` 与现有 `Capacitor` 原生壳。

边界：

- 它是过渡线，不是长期正式客户端主线。
- 后续不再以“所有新能力都必须先落在这条线”作为默认规则。
- 主要任务是稳住真机体验、存档、基础埋点和复盘闭环。

### 1.3 `Cocos Mainline`

定位：

- 未来唯一正式客户端主线。
- 用一条主线覆盖：
  - `H5`
  - `iOS`
  - `Android`
  - `小游戏`
  - `PC desktop`

边界：

- 未来正式联机功能优先落在这条线上。
- 未来正式品牌体验、动画、资源体系、平台能力与多端适配优先落在这条线上。
- 长期目标是让“正式网页版”从旧 HTML 页面迁移为 `Cocos H5`，但旧 `HTML Legacy` 仍独立保留。

### 1.4 `Referee Server`

定位：

- 未来联机版的服务端裁判与房间基础设施。
- 为 `Cocos H5 / iOS / Android / 小程序 / PC desktop` 提供统一的联机能力。

边界：

- 不参与离线单机版的启动与运行。
- 只为正式联机版本提供账号、房间、重连、回放和风控能力。

## 2. 路线图总原则

- 先稳住当前 `HTML / Capacitor` 手机版，再推进 `Cocos` 主线，而不是两边同时大改。
- 先抽 `game-core`，再做多端客户端；不要让渲染层继续反向绑定玩法层。
- 长期保留 `HTML Legacy`，但不要求它与 `Cocos Mainline` 在所有新功能上继续并线推进。
- 正式联机能力优先落到 `Cocos Mainline`，不再默认把旧 `HTML` 客户端扩成正式联机主战场。
- 联机阶段必须采用 `服务端权威判定`。
- 所有端都应共享同一套规则、AI、回放种子、开局码、日志结构和操作语义。
- 这也包括回放短码方案本身：当前 `回放种子 / 开局码` 已统一切到 `11` 位 / `169` 位字母数字混合格式，后续任何端都不能单独保留旧 hex 口径。

## 3. 需要删除的旧假设

以下旧假设从本路线图起不再适用：

- “当前 `Capacitor + WebView` App 会自然演进为长期正式主线客户端。”
- “未来所有新能力都默认先落在当前 `HTML` 牌桌上，再考虑其他端。”
- “账号与云存档一定先于 `Cocos` 单机主线落地。”
- “桌面端若要补上，默认继续沿用当前网页壳层而不是纳入正式主线。”
- “小游戏属于 App 主线之外的附属需求，后期单独再看。”

新的默认顺序是：

1. 稳住过渡期 `HTML / Capacitor`。
2. 抽离 `game-core`。
3. 建立 `Cocos Mainline` 单机版。
4. 以 `Cocos Mainline` 为基础推进联机。
5. 在 `Cocos Mainline` 上扩展小游戏与桌面端。

## 4. 目标仓库结构

建议逐步把仓库收敛到下面 5 个大模块：

- `packages/game-core`
  - 纯规则、状态机、AI、回放、seed、开局码、日志格式。
- `packages/game-protocol`
  - 客户端命令、服务端事件、快照、房间协议、重连协议。
- `apps/html-legacy`
  - 当前长期保留的无服务器 `HTML` 单机版。
- `apps/cocos-client`
  - `Cocos Mainline` 正式客户端。
- `services/referee`
  - 联机裁判、房间、账号与回放服务。

## 5. 分阶段路线

### Phase A：过渡期稳定与核心抽离

建议时间：`现在 - 6 周`

目标：

- 继续提升当前 `HTML / Capacitor` 真机体验。
- 为 `Cocos Mainline` 抽出可复用的 `game-core`。
- 把“旧牌桌继续发版”和“新主线开始搭骨架”拆成两件事处理。

核心范围：

- 持续修复当前手机端卡顿、白屏、手牌区布局和长局稳定性问题。
- 保持当前 `Capacitor` 打包、存储、复盘与基础埋点可继续交付。
- 把共享逻辑拆成：
  - `纯规则 / AI / 回放`
  - `混合状态机`
  - `纯 DOM 渲染`
- 定义统一的：
  - `GameCommand`
  - `GameSnapshot`
  - `ReplayRecord`
  - `ClientSettings`
- 明确旧 HTML 版今后只保单机主路径，不再为未来联机架构承担全部前置工作。

交付物：

- 一版仍可发布的 `HTML / Capacitor` 手机版。
- `game-core` 第一版接口草案。
- 继续可跑的 headless 回归。
- 文档同步后的多端路线基线。

验收口径：

- 当前 App 仍能稳定跑完整单机一局。
- `rules / AI / replay` 已能开始摆脱 DOM 依赖。
- `Cocos` 技术样机已具备明确输入协议，不需要再从 DOM 反推状态。

### Phase B：`Cocos` 技术样机

建议时间：`第 2-3 个月`

目标：

- 建立 `Cocos Mainline` 的最小可玩牌桌。
- 验证当前玩法核心能否稳定接入 `Cocos` 客户端。

核心范围：

- 新建 `apps/cocos-client`。
- 做最小竖切：
  - 牌桌背景
  - 5 个座位
  - 手牌区
  - 出牌区
  - 顶部状态栏
  - 基础点击出牌
- 只接 `snapshot -> 渲染`，先不接登录、广告、房间。
- 确认资源与牌面体系在 `Cocos` 下的组织方式。

交付物：

- 一版可在 `Cocos H5` 运行的最小单机样机。
- 一版可在 `iOS / Android` 真机启动的样机包。

验收口径：

- 能在 `Cocos` 样机里跑通最小单机流程。
- 手牌选择、出牌、阶段切换和结果页方向成立。
- `game-core` 不再依赖现有 DOM 结构才能运行。

### Phase C：`Cocos` 单机 MVP

建议时间：`第 3-5 个月`

目标：

- 做出第一版正式的 `Cocos` 单机客户端。
- 让未来正式 `H5 / iOS / Android` 客户端从这条线开始承接。

核心范围：

- 补齐完整单机流程：
  - 开始页
  - 发牌
  - 亮主 / 反主
  - 扣底 / 叫朋友
  - 出牌
  - 结算
  - 复盘输入
- 接入基础设置、牌面、日志导出与本地存档。
- 补齐 `Cocos H5 / iOS / Android` 的构建链路。
- 当前 `HTML / Capacitor` 线进入“低风险维护 + 少量必要修复”模式。

交付物：

- `Cocos H5` 单机版。
- `Cocos iOS / Android` 单机版。
- `HTML Legacy` 继续可静态部署、可离线运行。

验收口径：

- 新单机主线已能独立承接正式体验验证。
- 旧 HTML 版与 `Cocos` 版在规则、AI 和复盘输入上保持一致。
- 后续联机能力可以只接在 `Cocos Mainline` 上继续推进。

### Phase D：联机基础设施

建议时间：`第 5-8 个月`

目标：

- 建立正式联机底座。
- 让 `Cocos Mainline` 进入可建房、可补 AI、可重连的阶段。

核心范围：

- 建立 `services/referee`。
- 接入：
  - 账号主键
  - 云存档
  - 房间
  - WebSocket
  - 服务端权威裁判
  - AI 补位
  - 重连
  - 事件日志与回放
- 第一个联机客户端优先支持：
  - `Cocos H5`
  - `Cocos iOS / Android`

交付物：

- 联机 Alpha。
- 房间协议与服务端裁判第一版。
- 可用的联机回放链路。

验收口径：

- 2-5 人建房可用。
- AI 补位可用。
- 掉线重连与超时托管可用。
- 联机正式版本不再依赖旧 `HTML Legacy` 客户端。

### Phase E：小游戏版

建议时间：`第 8-10 个月`

目标：

- 在 `Cocos Mainline` 基础上输出小游戏版本。

核心范围：

- 适配：
  - 登录
  - 分享
  - 包体
  - 资源分包
  - 平台限制
  - 平台广告与合规能力
- 保持玩法、协议、回放和日志结构与主线一致。

交付物：

- 第一版小游戏测试包。

验收口径：

- 平台登录、进入房间、完整一局和回放链路跑通。
- 玩法表现与正式主线无规则偏差。

### Phase F：PC Desktop

建议时间：`第 10-12 个月`

目标：

- 在 `Cocos Mainline` 基础上补齐 `PC desktop` 版本。

核心范围：

- 输出 `Windows / macOS` 客户端。
- 补鼠标、键盘、大屏布局与窗口行为。
- 对桌面端单独收口：
  - 快捷键
  - 大屏密度
  - 结果页和日志面板
  - 安装、更新与桌面 QA

交付物：

- 第一版桌面测试包。

验收口径：

- 可完成完整对局。
- 大屏 UI 与鼠标输入体验达标。
- 与 `Cocos H5 / Mobile` 共用同一套核心规则与联机协议。

## 6. 近期 6 周拆解

### Week 1

- 冻结当前路线决策并完成文档同步。
- 盘点当前 `src/shared/` 中哪些文件可直接归入 `game-core`，哪些仍绑定 DOM。

### Week 2

- 抽第一批纯模块：
  - 规则
  - 牌型识别
  - 回放种子与开局码
  - AI 选择与评估入口

### Week 3

- 定义 `GameCommand / GameSnapshot / ReplayRecord`。
- 把旧牌桌渲染改成更多依赖快照，而不是直接散读全局 DOM。

### Week 4

- 继续优化当前 `HTML / Capacitor` 真机体验。
- 把 App 过渡版中最明显的渲染与交互卡顿先收口。
- 优先修掉 App 底部主操作条这类真机上会被高分屏放大的布局空白，尤其是按钮下方空轨和固定空列问题。
- 同步收口 App 手牌区牌面尺寸，让底部手牌和中央出牌区使用同一套卡牌口径，不再靠缩小手牌换空间。
- 把这套底部操作条和手牌尺寸修复同步到 `index2.html`，并在无主场景下补齐 5 行手牌与花色标签左对齐，避免 H5 / App 两端再次分叉。
- 继续把 `index2.html` 的常规出牌态和亮主/反主态拆开调优：前者允许主按钮更厚、更好点；后者必须保住亮主候选可见范围，同时把主牌行与花色行的牌轨起点也锁到同一条基线上。

### Week 5

- 初始化 `Cocos` 工程。
- 跑通资源加载、基础场景、手牌节点与桌面布局样机。

### Week 6

- 打通“从 `game-core` 读快照，在 `Cocos` 里渲染一轮最小对局”的演示。
- 评估 `Cocos` 样机在 `H5 / Android / iOS` 上的首轮可行性。

## 7. 各产品线功能落地优先级

### 7.1 `HTML Legacy`

优先保留：

- 单机完整玩法。
- 规则说明。
- 复盘输入与日志导出。
- QA 与 AI 调试入口。

不再默认优先新增：

- 正式联机。
- 新平台接入。
- 小程序适配。
- 正式长线运营系统。

### 7.2 `Cocos Mainline`

优先承接：

- 正式品牌与视觉。
- 新牌桌体验。
- 多端统一 UI。
- 正式联机。
- 小程序。
- 桌面端。

### 7.3 `Referee Server`

优先承接：

- 房间。
- 权威裁判。
- AI 补位。
- 重连。
- 回放。
- 举报与风控。

## 8. 质量守门

### 8.1 `HTML / Capacitor` 过渡线

- 单机完整一局通过率。
- 真机长局稳定。
- 原生存储可恢复。
- 复盘输入可复制与复现。

### 8.2 `Cocos Mainline`

- `H5 / iOS / Android` 三端单机一致性。
- 关键阶段 UI 与输入手感达标。
- 大局面不卡顿、不明显掉帧。

### 8.3 联机线

- 建房成功率。
- 完局率。
- 重连成功率。
- 非法操作拦截率。
- 回放可追溯率。

## 9. 风险列表

### 9.1 近期风险

- 旧 `HTML / Capacitor` 与 `Cocos Mainline` 双线并行，短期沟通成本会上升。
- 如果 `game-core` 抽离不够坚决，后续会把旧 DOM 假设继续带进 `Cocos`。
- 如果仍要求旧 HTML 线承接所有正式新功能，主线切换会失败。

### 9.2 中期风险

- `Cocos` 单机版如果迟迟不收口，联机会被旧 `HTML` 线继续拖住。
- 如果服务端协议在 `Cocos` 客户端稳定前就提前锁死，后续改动成本会变高。
- 小程序适配若过早开始，可能会稀释主线单机和联机的资源投入。

## 10. 决策优先级

如果资源有限，优先级固定为：

1. 稳住当前 `HTML / Capacitor` 手机版与离线单机价值。
2. 抽离 `game-core`。
3. 建立 `Cocos` 单机主线。
4. 做联机底座。
5. 扩小程序。
6. 补桌面端。

这条优先级不建议颠倒。
