# 五人找朋友 App、联机与广告运营路线图

这份路线图把项目从当前 Web 原型推进到 `移动 App -> 账号云存档 -> 好友房 -> 联机匹配 -> 长线广告运营` 的阶段目标统一整理下来。

如果没有额外资源变化，默认按“小团队、先验证留存、后扩大复杂度”的节奏推进。

配套文档：

- 产品需求见 [mobile-app-product-requirements.md](mobile-app-product-requirements.md)。
- 技术设计见 [mobile-online-architecture.md](mobile-online-architecture.md)。
- 首发任务拆解见 [mobile-app-launch-task-breakdown.md](mobile-app-launch-task-breakdown.md)。
- 联机接口与状态机草案见 [mobile-online-api-state-machine.md](mobile-online-api-state-machine.md)。

## 1. 路线图总原则

- 先做 `能稳定留人` 的单机 App，再做 `更复杂` 的联机。
- 先做 `好友房 + AI 补位`，再做 `公开匹配`。
- 先做 `轻广告验证`，再做 `广告优化与中介`。
- 先做 `日志与回放`，再做 `举报、风控、排位`。
- 单机阶段的日志基础设施要先把 `回放种子 + 开局码` 落稳，再往上叠完整事件流回放；否则后续客服、AI 调优与举报链路都会缺最小可复原输入。

## 2. 阶段总览

### Phase 0：准备期

目标：

- 固化 App 化与联机方案。
- 统一产品、设计、技术边界。
- 补齐构建、发布与测试基线。

交付物：

- App 产品需求文档。
- 联机技术设计文档。
- 本路线图。
- 构建与 QA 清单。
- `Capacitor` 工程初始化口径：`找朋友升级 / com.nolanli.cards / dist/app`。

### Phase 1：单机 App 首发

目标：

- 让当前项目作为 `iOS / Android` App 可稳定运行。
- 保留现有单机完整玩法与 AI 深度。
- 接入基础广告、埋点和崩溃监控。

核心范围：

- Capacitor 容器化。
- 原生存储替换浏览器持久化。
- 真机 UI 适配。
- 开局设置项跨端对齐：`AI难度 / 对局节奏 / 开始或继续游戏` 的操作语义保持一致，其中节奏入口统一使用四档按钮组而不是移动端单独下拉。
- 插屏与激励视频第一版。
- 公告、远程配置、埋点、崩溃监控。

验收口径：

- 新手能稳定完成完整一局。
- 继续游戏无丢档。
- 真机长局不白屏、不明显卡回合。
- 广告失败时不影响继续游玩。

### Phase 2：账号与云存档

目标：

- 建立用户主键。
- 为后续好友房、外观、战绩、联机和广告分群打底。

核心范围：

- Apple 登录。
- Google 登录。
- 游客绑定。
- 云存档同步。
- 多设备恢复。

验收口径：

- 换设备后能恢复等级进度和关键设置。
- 绑定流程不丢本地进度。

### Phase 3：好友房与 AI 补位

目标：

- 先把“熟人局”做通，而不是急着上公开匹配。
- 允许 2-5 人建房，空位由 AI 补齐。

核心范围：

- 建房 / 邀请码 / 入房。
- 座位分配。
- 开局准备。
- 掉线重连。
- 超时托管。
- 手动托管三态对齐：`关闭 / 本局托管 / 跨局托管`。
- 单机 / App 壳层里的结果弹窗继续做窄屏紧凑化，确保 6.1-6.7 英寸设备首屏能同时看清标题、级别结算、底牌亮出和主操作按钮。
- 单机 / App 壳层里的默认牌桌入口切到 `index-app.html`，并把牌桌收口成固定顶栏、固定底部、中央出牌区自适应，优先解决刘海屏和原生安全区对手牌空间的挤压。
- 服务端权威判定第一版。

验收口径：

- 房主能在 1 分钟内拉起一局。
- 断线后可回。
- AI 补位不影响完整结算。

### Phase 4：快速匹配与战绩

目标：

- 让没有固定牌友的玩家也能开局。
- 补齐联机长期运营所需的基础数据面板。

核心范围：

- 快速匹配。
- 最近同桌。
- 历史战绩。
- 举报与回放。
- 匹配失败 AI 补位。

验收口径：

- 匹配成功率可接受。
- 房间平均等待时长受控。
- 举报可追溯到回放。

### Phase 5：排位、赛季与运营优化

目标：

- 在联机稳定基础上引入更长线的目标感与变现优化。

核心范围：

- 段位与赛季。
- 排行榜。
- 活动任务。
- 广告中介。
- 外观系统。
- 长线留存运营。

## 3. 每阶段建议工期

以下工期按“小团队并行推进、共享规则层继续复用”的理想情况估算：

- `Phase 0`：1-2 周
- `Phase 1`：4-8 周
- `Phase 2`：3-5 周
- `Phase 3`：8-12 周
- `Phase 4`：6-10 周
- `Phase 5`：持续迭代

如果团队人数少于 3 名核心开发，整体周期需要进一步拉长。

### 4.2 更稳妥配置

## 5. Phase 1 详细拆解

### 5.1 技术工作

- 建立 App 打包工程。
- 首阶段先用 `Capacitor` 接入当前仓库，并固定 App Name 为 `找朋友升级`、App ID 为 `com.nolanli.cards`、`webDir` 为 `dist/app`。
- 统一资源打包与版本号。
- 清理 App Web 构建里的废弃脚本副本，并用单测锁住“最小资源清单”，避免历史 vendor 文件再次进入 iOS / Android 壳包。
- 把牌面资源链路统一成“经典逐张 SVG + 可生成的整图 sprite”双轨，确保 `m_cards_sprite.png` 这类单文件主题能直接进入 App 打包与后续热更新；但在重新通过视觉验收前，mobile 默认整图入口仍应固定为 `poker.png`。
- 把实验整图资源的切图与对齐规则纳入资源验收，确保未来若再次尝试切换默认牌面，不会在 App 小卡位里出现额外留边、偏移或花色错位。
- 持续压缩手游手牌区的非必要辅助信息，像花色标签张数这类 PC 专属读牌提示应优先移出 mobile 主视图。
- 把顶部 `主 / 朋` 这类状态牌也纳入移动端牌面验收，确保它们和底部手牌共用同一套主题与缩放口径，不再出现混合渲染。
- 替换浏览器式存档。
- 接入崩溃监控与埋点。
- 接入广告 SDK。
- 建立真机 smoke 流程。

### 5.2 UI 工作

- 首页 App 化。
- 局内顶部状态收紧。
- 顶栏高频入口补齐 `重置本局`：统一使用 `icon_new_game.png`，固定落在 `上一轮回看` 与 `设置` 之间，并明确保持“保级重发牌”的产品语义。
- 设置菜单继续做入口去重：`重置本局` 只保留顶部高频入口，不再在菜单里重复提供同义按钮。
- 手牌触摸优化。
- 设置页、结果页、规则页 App 化。
- 横竖屏与安全区适配。

### 5.3 QA 工作

- 弱网、断网、切后台、热启动、冷启动。
- 广告加载失败回退。
- 长时对局内存压力。
- 不同机型分辨率回归。

### 5.4 商店与发布工作

- 图标、截图、描述文案。
- 年龄分级。
- 隐私政策。
- 测试包分发。

## 6. Phase 2 详细拆解

- 抽象用户数据模型。
- 增加账号绑定弹层。
- 冲突存档合并策略。
- 埋点区分游客与登录用户。
- 做新旧存档迁移。

## 7. Phase 3 详细拆解

### 7.1 房间能力

- 房间创建。
- 邀请码入房。
- 房主权限。
- 开始前准备状态。

### 7.2 裁判层能力

- 服务端房间状态机。
- 合法性判定。
- 广播标准事件。
- 托管切换。

### 7.3 体验能力

- 断线重连。
- 长时间未操作提示。
- 房间内聊天可后置，不作为首批必须项。

## 8. 广告运营路线

### 8.1 首发阶段

- 以 `局间插屏 + 激励视频` 为主。
- 新用户前若干局免插屏。
- 激励视频仅用于外观与额外复盘奖励。

### 8.2 第一轮优化

- 根据留存与广告 ARPDAU 调整冷却。
- 识别高价值用户与重度用户，减少打扰。
- 对广告失败率高的机型做兜底。

### 8.3 第二轮优化

- 接入广告中介，提高 fill rate。
- 增加可配置原生广告位，但只放在首页或非对局页。
- 根据地区、时段、用户层级细分广告策略。

## 9. 多人在线路线

### 9.1 为什么先做好友房

- 五人局天然更适合熟人社交开局。
- 冷启动期公开匹配容易凑不齐人。
- 好友房更容易容纳 AI 补位。
- 更适合作为规则争议与回放体系的第一批真实样本。

### 9.2 为什么排位后置

- 排位会放大掉线、外挂、平衡、托管、公平性争议。
- 当前 AI、日志、回放虽已有基础，但还不适合直接承接排名压力。

## 10. 质量守门

### 10.1 App 版本守门

- 单机完整一局通过率。
- 切后台恢复成功率。
- 崩溃率。
- 广告展示成功率。

### 10.2 联机版本守门

- 建房成功率。
- 重连成功率。
- 完局率。
- 违规操作拦截率。
- 回放可用率。

## 11. 风险列表

### 11.1 近期风险

- 原型代码继续增长，导致 App 化时耦合过深。
- 广告过早过多，影响评分和留存。
- 没有账号主键，后续云存档迁移成本上升。

### 11.2 中期风险

- 联机裁判层抽离不彻底，客户端与服务端规则不一致。
- 房间状态与回放结构设计太晚，导致排查困难。
- 匹配过早上线，等待时间和 AI 补位体验失衡。

## 12. 里程碑建议 KPI

### 12.1 App 首发阶段

- 首局完成率达到可接受水平。
- `D1 / D7` 留存稳定。
- 广告展示不明显压低首局完成率。

### 12.2 好友房阶段

- 好友房开局成功率高。
- 平均组局耗时可控。
- AI 补位接受度高。

### 12.3 匹配阶段

- 平均等待时长可控。
- 完局率与举报率健康。
- 回放和风控可以支撑客服处理。

## 13. 决策优先级

如果资源有限，优先级固定为：

1. App 化与真机稳定。
2. 存档、埋点、监控、广告基础设施。
3. 账号与云存档。
4. 好友房与 AI 补位。
5. 快速匹配。
6. 排位与赛季。

这条优先级不建议颠倒。

## 14. 当前建议的最近行动

最接近落地的下一步建议如下：

1. 把当前项目补成稳定的 App 构建与发布流程。
2. 把浏览器存档迁移策略确定下来。
3. 把共享规则层继续去 DOM 依赖，为未来服务端裁判做准备。
4. 先出真机 UI 稿与广告位稿。
5. 再开始接入原生壳与基础 SDK。

后续如果需求变化，这份路线图需要和产品需求、技术设计文档一起同步更新。
