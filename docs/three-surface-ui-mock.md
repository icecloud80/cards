# 三端主战场界面图说明

这份文档记录 `PC / H5 / App` 三端主战场的中保真界面图、统一视觉方向和实现边界，作为后续静态模板、真实页面改造和评审截图的共同参照。

配套产物：

- PC 设计稿：`images/mockups/ui-mock-pc.svg`
- H5 设计稿：`images/mockups/ui-mock-h5.svg`
- App 设计稿：`images/mockups/ui-mock-app.svg`

## 1. 目标

- 为当前项目补一套可直接评审的三端界面图，不再只停留在文字计划。
- 明确一期重点是 `主战场布局 + 操作效率 + 响应速度 + 视觉清爽化`。
- 给后续 `index1.html / index2.html / index-app.html` 的改造提供统一参照，避免三端分别长成三套产品。

## 2. 统一视觉方向

- 核心气质统一为 `现代清爽桌游感`。
- 主背景继续保留牌桌氛围，但整体收口成 `深青绿 + 雾面玻璃 + 金色点缀 + 蓝色主操作`。
- 金色只用于主牌、结果和高优先级强调，不再把大量说明和按钮都做成高亮。
- 面板统一更轻、更薄，减少“原型式厚面板”的堆叠感。
- 关键动作统一靠近手牌区，降低视线来回跳转成本。

## 3. 三端差异

### 3.1 PC

- 布局改成 `左侧轻状态列 + 中央对称战场 + 底部连续手牌轨 + 右侧焦点操作区`。
- 重点是让玩家先读局面，再读操作，而不是先看到大量菜单和文字说明。
- 适合继续承载更高信息密度，如玩家状态摘要、推荐操作和更清晰的牌型展示。

### 3.2 H5

- 布局改成 `极简顶栏 + 中央 5 席位 + 固定手牌托盘 + 底部主操作条`。
- 优先处理浏览器顶底栏带来的可视区域压缩问题。
- 继续保留分组手牌，但弱化说明文案和低频入口，让阅读路径更短。

### 3.3 App

- 布局固定为 `Safe Area 顶栏 + 中部自适应战场 + 固定手牌托盘 + 固定拇指操作条`。
- 拇指操作条要保持单行薄轨，安全区留白由外层容器统一吸收；按钮条本身不能再额外吃一层底部安全区，否则会把最后一行手牌顶住。
- 拇指操作条里的按钮行要跟随当前可见按钮数自动均分，按钮底边也应尽量贴住容器底部，避免高 DPR 手机把底部空白放大成明显的 ghost gap。
- 对于 `index2` 这类浏览器运行页，`选择 / 出牌` 的常规主按钮可以做成更厚的单行胶囊，但 `亮主 / 反主` 候选区要继续保持紧凑，不能被同一套放大规则一起往屏幕外挤。
- 手牌托盘里的牌面尺寸应和中央出牌区保持同口径，只通过重叠量而不是缩小牌面来换空间。
- 无主场景下，`主牌 + 四门副牌` 应优先展开成 5 条左对齐手牌轨；不要为了省高度把某一门花色挤到主牌同一行右侧，导致花色标签错位。
- 当前移动端实现口径补充为：`index2` 与 `App` 的手牌区都要在上一版基础上再增加 `10px`；底部操作区也重新统一到同一套高度，避免两端在相同按钮语义下却出现不同的触达肌肉记忆。
- `主牌` 行和下面花色行的第一张牌也必须沿同一条竖线起步，避免看起来像只有标签对齐、牌轨本身还在错位。
- 顶部 `主 / 朋` 小卡位统一维持“纯牌面 + 短标签 + 角标”视觉，不再额外套橘色描边；可点击态也不应因为强调框而破坏主牌与朋友牌的一致口径。
- 关键阶段统一改成步骤浮层，避免继续把说明文字分散在多个角落。
- 和 H5 共享整体设计语言，但在安全区、底栏和面板高度上明确区分。

## 4. 操作与性能目标

- 倒计时刷新不应再带动整屏 UI 大面积重建。
- 手牌选中、撤销、出牌反馈必须比当前更直接。
- 顶部状态、焦点步骤卡、中央播报和底部主按钮应形成稳定的四段阅读路径。
- H5 与 App 的移动端壳层应尽量复用同一套渲染适配逻辑，不继续复制粘贴两份近似脚本。

## 5. 规则、玩法、AI 与 Heuristic 边界

- 这次界面图不改规则。
- 这次界面图不改玩法逻辑。
- 这次界面图不改 AI 策略。
- 这次界面图不改 AI heuristic。
- 这次界面图只调整表现层、交互层和渲染层的目标方向。

## 6. 路线图关系

- 这套界面图服务于 `Phase 1：单机 App 首发` 的 UI / UX 落地。
- 它优先解决“当前项目像原型而不像产品”的观感问题。
- 它也为后续真机截图、商店素材、静态模板和 UI smoke 提供共同母稿。

## 7. 风险与 bug 关注点

- 如果只改视觉不改渲染分区，倒计时和 AI 等待阶段仍可能继续卡顿。
- 如果 PC / H5 / App 继续各自独立修样式，后面会再次出现跨端术语和操作顺序分裂。
- 如果底部主操作按钮不固定位置，移动端误触和学习成本会继续偏高。

## 8. 文档同步要求

- 产品需求文档要同步引用这套界面图。
- PC 设计文档要同步引用这套界面图。
- 路线图要同步补上“先出三端主战场母稿”的里程碑。
- 最近更新记录和 bug fix 列表要记录这次新增的设计稿资产。

## 9. 后续落地建议

- 先把这三张 SVG 作为评审稿确认方向。
- 再把它拆成 `PC 静态模板 / H5 静态模板 / App 静态模板`。
- 最后再做共享渲染分区和真实页面替换，避免一上来就把视觉与逻辑重构绑死。
