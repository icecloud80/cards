# JS 注释模版：帮助人理解 AI 写的代码

这份模版的目标不是解释“代码做了什么表面动作”，而是解释：

- 为什么这样写
- 输入和输出是什么
- 哪些地方容易改坏
- 读代码的人接下来该看哪里

## 1. 文件头注释模版

适合放在模块、工具函数文件、复杂业务文件顶部。

```js
/**
 * 这个文件负责什么：
 * - [一句话说明核心职责]
 *
 * 业务背景：
 * - [这个文件为什么存在]
 * - [它解决的具体问题]
 *
 * 核心流程：
 * 1. [第一步]
 * 2. [第二步]
 * 3. [第三步]
 *
 * 输入：
 * - [外部依赖、参数、状态、配置]
 *
 * 输出：
 * - [返回值、状态变化、副作用]
 *
 * 关键约束：
 * - [不能破坏的规则]
 * - [性能、顺序、幂等、兼容性要求]
 *
 * 修改时注意：
 * - [最容易出 bug 的点]
 * - [改动后建议验证什么]
 */
```

## 2. 函数注释模版

适合核心函数、业务函数、名字不够自解释的函数。

```js
/**
 * 作用：
 * [一句话说清这个函数的目标]
 *
 * 为什么存在：
 * [为什么不能直接写在调用处，或为什么需要单独封装]
 *
 * 参数：
 * @param {Type} input - [这个参数在业务里的含义]
 * @param {Type} options - [可选配置会影响什么]
 *
 * 返回：
 * @returns {Type} [返回结果的业务含义]
 *
 * 处理逻辑：
 * 1. [先做什么]
 * 2. [再做什么]
 * 3. [最后产出什么]
 *
 * 边界情况：
 * - [空值、异常值、默认值]
 * - [哪些输入会提前返回]
 *
 * 不要轻易改动：
 * - [某段判断/排序/节流/缓存的原因]
 */
function example(input, options) {
  // ...
}
```

## 3. 复杂代码块注释模版

适合条件分支、排序逻辑、状态同步、数据转换、缓存命中等“看得懂但不容易一下理解为什么”的代码。

```js
// 这里先 [动作]，不是为了省一步，而是为了保证 [原因]。
// 如果直接 [另一种写法]，会导致 [具体问题]。
[复杂代码块]
```

## 4. AI 代码最值得补充的注释类型

如果是 AI 生成的代码，优先补下面 4 类注释，而不是逐行翻译代码：

```js
// 业务意图：这段逻辑是在防止重复提交，而不只是“判断 loading”。

// 设计原因：这里用 Map 而不是对象，是因为 key 需要保留原始类型。

// 隐含约束：返回数组顺序必须稳定，因为 UI 是按索引做动画的。

// 修改风险：这里和服务端字段名强耦合，改之前要确认接口版本。
```

## 5. 推荐的短注释句式

需要写得短一点时，可以直接套这些句式：

- `这里先做 [X]，是为了 [Y]。`
- `不要直接改成 [方案]，否则会影响 [行为]。`
- `这个判断不是多余的，它在处理 [边界场景]。`
- `这里依赖 [前置条件]，调用方必须保证成立。`
- `这里返回 [结构]，方便下游直接用于 [用途]。`
- `顺序不能变，因为 [原因]。`
- `这里故意不抛错，而是返回默认值，用来保证 [体验/稳定性]。`

## 6. 实战示例

```js
/**
 * 作用：
 * 把搜索结果整理成前端列表可直接渲染的数据结构。
 *
 * 为什么存在：
 * 服务端返回的数据字段比较分散，这里统一做一次映射，
 * 避免组件层重复处理同样的转换逻辑。
 *
 * 参数：
 * @param {Array<Object>} items - 服务端原始搜索结果
 * @param {string} keyword - 当前搜索关键词
 *
 * 返回：
 * @returns {Array<Object>} 可直接渲染的列表项
 *
 * 边界情况：
 * - 空数组时直接返回 []
 * - title 缺失时回退为 "Untitled"
 *
 * 不要轻易改动：
 * - 返回顺序必须和原始结果一致，否则高亮定位会错位
 */
function buildSearchList(items, keyword) {
  if (!Array.isArray(items) || items.length === 0) {
    return [];
  }

  // 这里保留原始顺序，因为界面高亮是按索引同步的。
  return items.map((item, index) => {
    const title = item.title || "Untitled";

    // 关键词命中信息提前算好，组件层只负责展示，减少模板里的判断复杂度。
    const matched = keyword ? title.includes(keyword) : false;

    return {
      id: item.id,
      index,
      title,
      matched,
    };
  });
}
```

## 7. 最实用的极简版模版

如果你只想要一个平时最常用的版本，用这个：

```js
/**
 * 作用：
 * [它做什么]
 *
 * 为什么这样写：
 * [设计原因 / 业务原因]
 *
 * 输入：
 * @param {Type} xxx - [业务含义]
 *
 * 输出：
 * @returns {Type} [业务含义]
 *
 * 注意：
 * - [边界条件]
 * - [不要随便改的点]
 */
```

## 8. 判断注释有没有价值的标准

一条好注释，至少回答下面一个问题：

- 这段代码为什么存在？
- 为什么不用更直觉的写法？
- 改这里会影响什么？
- 这段逻辑依赖什么前提？
- 调用方应该怎么安全使用它？

如果注释只是在重复代码表面含义，例如：

```js
// 遍历数组
items.forEach(...)
```

这种通常可以删掉。
